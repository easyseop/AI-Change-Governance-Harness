## 🔴 R-1 (보정 사유): `continue-on-error` 우회 감지가 정확 문자열 `true` 만 매칭 → 자명한 YAML boolean 변형으로 집행 무력화 통과

**근원**: `detect_enforcement_bypass` (line 359)

```python
added_bypass_lines = [line.strip() for line in added
                      if "|| true" in line or "continue-on-error: true" in line]
```

`continue-on-error: true` **정확 부분문자열**(소문자·단일공백·무따옴표)만 잡는다. 그러나 GitHub Actions(YAML 1.2 core schema)는 `True`·`TRUE`·`"true"`·`yes`·`on` 및 여분 공백을 **동등한 boolean true 로 수용**한다 → 아래는 전부 스텝 실패를 무시(집행 무력화)하지만 게이트는 **pass**.

**fresh 실증(픽스처 밖, 게이트 호출 라인은 무변경·`continue-on-error` 만 새 라인 추가)**:

| head 삽입 라인 | 실제 GHA 효과 | 게이트 verdict |
|---|---|---|
| `continue-on-error: true`  | 집행 무력화 | **approval_required** ✓ |
| `continue-on-error: True`  | 집행 무력화 | **pass** ✗ 우회 |
| `continue-on-error: TRUE`  | 집행 무력화 | **pass** ✗ 우회 |
| `continue-on-error:  true` (2칸) | 집행 무력화 | **pass** ✗ 우회 |
| `continue-on-error: "true"` | 집행 무력화 | **pass** ✗ 우회 |

- **위반**: AC #7(b) "`|| true`·`continue-on-error` 류 **무력화 삽입**을 감지". 정준형만 잡고 자명한 등가 변형을 놓치면 감지가 성립 안 한다. TASK-018 은 **P0 자기무력화 방지**("안 막으면 나머지 통제 전부가 무의미") — 집행 스위치를 끄는 변경이 통과하면 태스크 목적이 직접 뚫린다.
- **§2B 필수 질문("거버넌스 목적에 직접 구멍?") → 그렇다.** 비차단 불가.
- **AC 제약과의 정합**: `True`/`TRUE`/`"true"`/공백 정규화는 여전히 **결정적 패턴 매칭**이다 — "의미 추정 금지"(AC #7)를 위반하지 않는다. 오히려 등가 boolean 을 결정적으로 정규화해야 AC #7(b) 를 실제로 충족한다.
- **수정 계약**:
  - `continue-on-error` 값이 truthy 인지 **결정적으로** 판정: 키를 정규화(공백 무시)하고 값을 소문자화해 `{true, yes, on}`(따옴표 유무 무관) 와 매칭. (YAML boolean 진리값 집합 — 확정적, LLM/추정 아님.)
  - `|| true` 도 같은 취지로 최소한 공백에 견고하게(예: `||\s*true\b`) — 단, `|| :`/`|| exit 0` 등 다른 shell 무력화는 이미 removed-invocation 경로로 안전측 포착되므로 R-1 의 핵심은 `continue-on-error` 다.
  - **회귀 픽스처(R-2 와 함께)**: `continue-on-error: True`(또는 대문자/따옴표 변형) 삽입 → approval 을 **고정** + 음성검증(정규화 제거 시 FAIL).

## 🟠 R-2 (보정 사유): AC #7 픽스처 미완 — (b)`continue-on-error` 삽입·(c)required-check 제거 회귀 픽스처 부재
