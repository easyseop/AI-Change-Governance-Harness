## 1. 🔴 R-1 (차단 · `main` 보다 나쁨 · 흔적 없는 과소탐)

### 증상
소음을 만드는 그 조건(= sink 폐쇄에 평범한 JDK 호출 1개)이 **탐지 fallback 을 꺼버린다.** `check-indirect-impact.py` 의

```python
or not opaque_receiver_types      # ← "전부(all) 타입 미상" 일 때만 보수 승격
```

는 **전칭(all)** 판정이다. 불투명 폐쇄에 **타입을 붙일 수 있는 호출이 하나라도** 섞이면(`rows.size()` 한 줄이면 충분) 이 분기가 죽고, **타입을 붙일 수 없는 진짜 dispatch 는 아무 흔적도 남기지 못한 채 드롭**된다. 필요한 것은 **존재(any) 판정** — *"타입을 모르는 불투명 호출이 하나라도 있으면 보수"*.

Codex 자신의 요약도 *"**타입 미상**·불투명 조회·합성 초기화는 보수 판정을 유지한다"* 라고 적었다 — **의도는 맞고 구현이 전칭/존재를 뒤집었다.**

### 픽스처 밖 fresh 실증 (별도 임시 repo · `sensitive-zones-empty` · sink=`Flow.sink` · 변경=`Vault.transfer` 본문)

### 왜 비차단이 아닌가 (§2B 필수질문 = **예**)
P4 브랜치 출력은 `verdict: pass` · `exit 0` · `errors: []` · `fail_closed: []` · `indirect_impact: []` — **완전히 깨끗한 통과**다. coverage 에 `receiver_type` 없는 레코드가 남지만 **판정·카드 어디에도 신호가 없다.** A-0051 R-4 와 같은 "흔적조차 없는 과소탐" 등급이고, MVP-3 계약(**과탐 반올림·과소탐 금지**) 정면 위반이다.

### 근본원인 — 안전 방향이 뒤집힌 헬퍼 재사용
`receiver_type()`(`extract-java-callgraph.py:290`)은 원래 **호출 대상 탐색을 넓히려고** 만든 헬퍼라 체인 표현식의 **루트 식별자**로 추측한다(`re.split(r"[.(]", text, 1)[0]`). 틀려도 예전엔 대상 탐색이 실패해 `unresolved` 로 **안전하게 떨어졌다**. 이번 변경이 그 값을 **보수 승격을 좁히는 근거**로 쓰면서 **오류의 안전 방향이 뒤집혔다** — 이제 잘못된 추측은 **경보를 지운다**.

실측 2건:
- **일반타입 누출**: `List<String> rows` → `receiver_type: "String"` (컨테이너가 아니라 **타입 인자**). N4 픽스처 기대값에 그대로 박혀 있다.
- **체인 오귀속**: `holder.task.run()` → `receiver_type: "Holder"` (실제 수신자 `holder.task` 의 선언 타입은 `Runnable`). **P7 이 뚫리는 직접 원인.**

### (a) 추출기 — 정직한 수신자 타입 (`extract-java-callgraph.py`)
`receiver_type()` 은 **건드리지 말 것**(엣지 탐색의 넓힘 근거라 좁히면 엣지를 잃는다). 대신 **레코드 기록 전용** 헬퍼를 새로 두고, `method_invocation` 의 `unresolved` 레코드에만 쓴다:

```python
def precise_receiver_type(source_bytes, object_node, bindings, current_owner):
    """즉시 수신자의 선언 타입. 정확히 못 잡으면 None(=미상)."""
    if object_node is None:                      # 무자격 호출 = this
        return simple_name(current_owner)
    if object_node.type == "object_creation_expression":
        return receiver_type(source_bytes, object_node, bindings, current_owner)
    text = node_text(source_bytes, object_node).strip()
    if object_node.type == "this":
        return simple_name(current_owner)
    if object_node.type == "identifier":
        if text in bindings:
            return bindings[text]
        return simple_name(text) if text[:1].isupper() else None
    if object_node.type == "field_access" and text.startswith("this."):
        return bindings.get(text.split(".", 2)[1])
    return None      # ← 체인(field_access·method_invocation·array_access·cast…) = 미상
```

## 7. 🔴 자기정정 — 내 AC#1 이 이 함정을 유도했다

AC#1 은 축을 *"미해소 호출의 **수신자 선언 타입** ↔ 지연 레코드의 대상 타입"* 으로 지목하면서 **"수신자 타입을 못 구하면 어떻게 하라"** 를 안 적었다. 그래서 **미상 처리(전칭 vs 존재)** 가 구현자 재량으로 남았고 안전하지 않은 쪽이 선택됐다. 또 나는 *"착수 전 필수 fresh 검증축 ①~⑧"* 을 걸면서 **⑥ 불투명 dispatch 표준형에 "소음과 *동시에* 존재하는" 조합을 명시하지 않았다** — 그 조합이 정확히 P4/P7 이다.

**교훈(A-0051 §"판별식을 좁히는 변경" 의 확장)**: **기존 헬퍼를 정밀화 근거로 재사용하면, 그 헬퍼가 원래 어느 방향으로 틀렸는지를 반드시 다시 따져라.** 넓히는 데 쓰던 근사(`receiver_type`)는 틀려도 안전했지만, 좁히는 데 쓰면 같은 오차가 곧 놓침이 된다. 그리고 **미상값의 처리 규칙은 AC 에 못박아라**(존재-판정 = 보수).
