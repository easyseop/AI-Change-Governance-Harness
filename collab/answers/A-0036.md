## 1. Q1 — 공유 파일 확장 vs 별도 카탈로그 → **별도 파일** (실측 근거)

**답: `policies/java-sensitive-capabilities.yaml` 신규 생성. `policies/sensitive-capabilities.yaml` 은 수정하지 않는다.**

이건 취향 문제가 아니라 **실측된 회귀 방지 요건**이다.

### 근거 (코드 경로)

`.harness/gates/extract-python-capabilities.py:66-69`

```python
signals = raw.get("signals") or {}
known = {"imports", "calls", "builtins"}
for key in sorted(set(signals) - known):
    errors.append({"error": "unknown_signal_kind", "id": cap_id, "kind": key})
```

- 이 추출기는 `capabilities:` 목록을 **언어 필터 없이 전부** 읽는다.
- `{imports, calls, builtins}` 밖의 신호 키는 **검증오류**가 된다.

`.harness/gates/check-new-capabilities.py:258`

```python
if new_capabilities or fail_closed or errors:
    verdict = "approval_required"
    exit_code = APPROVAL_REQUIRED
```

- 카탈로그 오류는 **경로별로 누적**(`:224-227`)되어 **errors 하나만 있어도 approval_required**.

### 실증 (fresh repo · 픽스처 밖)

무해한 2줄 Python PR(`def sub` 추가)로 A/B:

```
=== BENIGN PR + 현행 공유 정책 ===
verdict: pass               exit: 0   errors: 0
=== BENIGN PR + Java 항목 주입 정책 ===
verdict: approval_required  exit: 2   errors: [
  {"error":"unknown_signal_kind","id":"command_exec","kind":"method_chains","path":"calc.py","side":"base"},
  {"error":"unknown_signal_kind","id":"command_exec","kind":"types","path":"calc.py","side":"base"}]
```

⇒ **Java 를 한 줄도 건드리지 않은 순수 Python PR 이 승인요구로 뒤집힌다.**
이는 D-074·D-076 이 계속 피해온 "환경·정책 사정으로 전 PR 승인요구" 실패모드 그 자체다
(= 게이트 신뢰 붕괴 → 사람이 무시 → 거버넌스 무력화).

### 부수 이득

- Python 추출기 **무개조**. 이미 통과·배포된 층에 손대지 않으므로 **회귀 위험 0**.
  (언어 필터를 추가하는 대안은 shipped 게이트 수정 = 재검증 비용 + 회귀 위험.)
- **선례 일치**: J2 의 `framework-annotations.yaml` 도 같은 이유로 분리 파일이다.
- **오배선은 시끄럽게 실패**한다 — Java 카탈로그를 Python 추출기에 잘못 먹이면
  `errors: 11 (kinds: methods, types)` 로 즉시 approval_required. 조용한 오통과 아님. (그래도 배선하지 마라 → AC#10)

## 2. Q2 — Java capability 레코드 확정

8개(AC#1 의 7종 + `crypto` 분할). 전부 `protected`, 단 `crypto_hash` 만 `watched`.

| id | level | Python 대응 | 근거 |
|---|---|---|---|
| `command_exec` | protected | `subprocess_exec`(protected) | CWE-78 |
| `unsafe_deserialization` | protected | `unsafe_deserialization`(protected) | CWE-502 |
| `reflection` | protected | `dynamic_code_exec`(protected) | CWE-470 |
| `sql_injection_surface` | protected | (신규) | OWASP A03 · CWE-89 |
| `jndi_lookup` | protected | (신규) | CWE-74 · Log4Shell 계열 |
| `outbound_http` | protected | `outbound_network`(protected) | OWASP A10 · CWE-918 |
| `crypto` | protected | `crypto_primitive`(protected) | CWE-327·321 |
| `crypto_hash` | **watched** | (Python 은 `hashlib` **의도적 제외**) | CWE-328 |

`reviewer`·`owner` 는 전부 `security-reviewer`, `source` 는 `builtin: cwe|owasp`. `maturity` 는
미지정(추출기 기본 `enforcing`) — Python 카탈로그와 동일하게 둔다.

### 🔴 AC#1 로부터의 의도적 편차 1건 — `MessageDigest` 분리

AC#1 은 "`Cipher`/`MessageDigest`(crypto)" 로 **한 항목**을 지시했으나 **둘로 나눴다**.

- Python 카탈로그는 `cryptography`·`Crypto`·`ssl`·`hmac` 을 protected 로 걸면서
  **`hashlib` 은 명시적으로 넣지 않는다**(잡음).
- `MessageDigest.getInstance("SHA-256")` 은 체크섬·ETag·캐시키 등 무해한 용도가 압도적 —
  Python 의 `hashlib` 과 정확히 같은 위치다.
- 이걸 protected 로 두면 **Java 만 Python 보다 엄격** = parity 왜곡(AC#6 의 "동일 verdict" 를
  스스로 깨는 방향) + 잡음 증가.
- ⇒ 능력은 카탈로그에 **남기되**(AC#1 목록 충족·카드 노출 유지) **등급만 `watched`**(경고, 승인요구 아님).
- `Mac`(=HMAC)은 Python `hmac` 이 protected 이므로 `crypto` 쪽에 뒀다(대칭 유지).

AC#1 이 "등급은 정책값"이라 위임했으므로 정책 소유자 권한 내 결정이나, **AC 문구와 다르므로 명시 기록**한다.
이견 있으면 재논의 — 조용히 바꾸지 않는다.

### 이연 2건 (조용한 누락 아님 — 파일에 기록)

- **인증서 검증 우회**: Java 대응은 커스텀 `TrustManager`/`HostnameVerifier` **구현**이라
  호출이 아닌 **`implements:` 신호**가 필요 → 아래 §3 신호 스키마 밖 + AC#1 목록 밖.
  스키마 확장은 추출기 계약 변경이므로 별도 태스크로 분리한다.
  (Python 은 `ssl._create_unverified_context` 로 이걸 "특히 위험"으로 잡고 있어 **parity 갭으로 남는다** — 인지하고 이연.)
- **자격증명·PII**: Python 카탈로그와 동일 사유(값·데이터흐름 분석 필요)로 이연.

## 3. Q3 — 신호 스키마 확정 (Java 는 Python 과 대칭이 아니다)

**신호 4종**: `imports` · `types` · `calls` · `methods`. (Python 의 `builtins` 는 Java 에 없음.)

| 종류 | 매칭 규칙 |
|---|---|
| `imports` | import 선언 FQN. **자기 자신 또는 하위 접두** 일치. `javax.crypto` → `import javax.crypto.Cipher;`·`import javax.crypto.*;` 둘 다 |
| `types` | 타입 **simple name**. (a) `new X(...)` (b) 정적 수신자 `X.m(...)` (c) 선언 타입 `X v = ...`. **마지막 점-세그먼트 기준** 매칭 → `new java.lang.ProcessBuilder()` 도 잡힘 |
| `calls` | 정규화 `Type.method`. 체인의 **구문상 뿌리**가 타입이름 또는 `new X(...)` 이면 그 simple name 이 Type |
| `methods` | **수신자 무관** 메서드명. 뿌리가 변수라 타입 해소 불가한 경우의 최후 백스톱 |

### Q3 가 지목한 4개 케이스에 대한 답

**(a) 암시적 `java.lang` — `Runtime.getRuntime().exec`**
→ import 문이 **없으므로** import backstop 이 원리적으로 안 통한다(TASK-032 AC#2 의 비대칭).
`calls` 로 커버: 체인 뿌리 `Runtime` 이 타입이름 ⇒ `Runtime.getRuntime` · `Runtime.exec` 두 신호 생성.
`ProcessBuilder`·`Class` 도 동일.

**(b) import backstop — `java.io.ObjectInputStream`**
→ 명시 import 이므로 `imports` 유효. **그리고 이게 진짜 중요한 이유**:
`ObjectInputStream ois = new ObjectInputStream(in); ois.readObject();` 에서
`ois` 는 **변수**라 `calls` 정규화로는 타입을 알 수 없다(타입추론 = 값 추정 = 불변식 3 위반).
⇒ `imports`(FQN) + `types`(`new ObjectInputStream`) + `methods`(`readObject`) **3중 백스톱**으로 메운다.
계층 방어가 설계 의도다 — 한 층이 뚫려도 다른 층이 잡는다.

**(c) 메서드 체인 — `new ProcessBuilder(...).start()`**
→ 뿌리가 `new X(...)` 이면 Type = `X`. ⇒ `ProcessBuilder.start`.
`Class.forName(n).getMethod(m)` → `Class.forName` · `Class.getMethod`.

**(d) 리플렉션·미해소 마커**
→ **추정 금지**(불변식 3). 해소 불가 동적 호출은 판정에 쓰지 말고
`unresolved_dynamic` 으로 **coverage 에 정직 노출**(`defaults.unresolved_dynamic: expose`).
단 `Class.forName`·`Method.invoke` 처럼 **구문상 확정되는** 리플렉션은 `reflection` 능력으로 정상 감지한다.

### `methods` 신호 사용 제한 (잡음 관리)

`methods` 는 4종 중 **과탐 위험이 가장 크다**. "잡음 폭증 = 게이트 무시 = 거버넌스 무력화"는
설계 §3 이 명시한 실패모드다. ⇒ **이름 자체가 고신호인 것만** 넣었다: `exec` · `readObject` · `lookup`.
`invoke` 는 프레임워크 관용구에서 흔해 **의도적으로 제외**했다(대신 `Method.invoke` call 신호 + `unresolved_dynamic`).

## 4. Q4 — frozen 금지·protected clamp → **확인. 그리고 코드 하드 플로어로 강화한다.**

**확인**: Java 카탈로그 level 은 `protected|watched` 만. `frozen` 이 오면 **검증오류 기록 + `protected` clamp**
(조용히 무시 금지). 근거: 능력 감지는 **2층(추론)** 이고 2·3층은 자동 차단 금지 — CLAUDE.md §4 · D-004.

**단, 그냥 확인만 하면 D-077 O-2 를 반복한다.** J2 리뷰에서 관찰한 것:
clamp 기준을 **정책 파일 자신의 `allowed_levels`** 에서만 읽으면 **정책을 고쳐 불변식을 끌 수 있다**.
불변식이 정책값으로 껐다 켜지면 그건 불변식이 아니다.

⇒ **TASK-032 AC#7 로 명시 가드화**(비차단 관찰로 흘리지 않음 — CLAUDE.md §2B):
게이트 **코드에 하드코딩된 상한**을 두고 정책값은 *더 좁히는* 방향만 허용.
검증은 `level: frozen` **+** `allowed_levels: [frozen]` **동시 rig** 로 — 한쪽만 rig 하면 이 결함을 못 잡는다.
