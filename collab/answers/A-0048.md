## 2. 🔴 R-1 (차단) — 승격 정밀화가 JDK 함수형 인터페이스에서 fail-closed 를 통째로 끈다

### 2.1 결함

`check-indirect-impact.py` 의 판별식이 "막다른 길"을 **`java_graph["nodes"]` 중 `bodyless` 인 것** 으로만 정의한다.

```python
java_bodyless = {n["id"] for n in java_graph.get("nodes", []) if n.get("bodyless") and n.get("id")}
sink_dead_ends = sink_reachable_bodyless_functions(sinks, adjacency, java_bodyless)
if java_deferred and sink_dead_ends:   # ← 여기
```

`nodes` 는 **repo 안에서 선언된 메서드**만 담는다. 따라서 dispatch 대상 인터페이스가 **JDK/외부 라이브러리**면 (`Runnable`·`Callable`·`Consumer`·`Function`·`ExecutorService`) 노드가 아예 없고 → `sink_dead_ends` 가 **빈 집합** → `java_deferred` 가 아무리 많아도 **fail-closed 가 발동하지 않는다**.

즉 정밀화가 "sink 와 무관한 dispatch 를 거른다" 가 아니라 **"dispatch 대상이 repo 밖이면 판정을 포기한다"** 로 동작한다. 실 Java repo 의 함수형 인터페이스는 압도적으로 JDK 쪽이므로 **L3 fail-closed 의 대부분이 꺼진다.**

## 3. 🔴 R-2 (차단) — AC#1 이 `class_body` 직속 자식에서만 닫혔다 (enum·인터페이스 상수·익명 본문)

`collect_initializer_deferred` 의 순회가 **타입 선언의 `body` 직속 named_children** 만 본다.

```python
for child in body.named_children:
    if child.type == "field_declaration": scan_deferred(child, f"{owner}.<init>")
    elif child.type == "static_initializer": scan_deferred(child, f"{owner}.<clinit>")
    elif child.type == "block": scan_deferred(child, f"{owner}.<init>")
```

tree-sitter-java 에서 **enum 멤버는 `enum_body_declarations` 아래**, **인터페이스 멤버는 `constant_declaration`** 이라 둘 다 이 필터를 통과하지 못한다. 익명클래스 본문(`class_body` under `object_creation_expression`)은 `TYPE_DECLARATIONS` 가 아니라 `visit` 자체가 들어가지 않는다.

**X13 ↔ X13b 가 원인을 못박는다**: 로직·정책·sink 가 전부 같고 **컨테이너 종류만** 다른데 판정이 갈린다 ⇒ 원인은 순회 형태이며, AC#1 이 지목한 결함과 **동일한 결함**이다. AC#1 문언은 컨테이너를 한정하지 않았고("AC 문언은 위치를 한정하지 않는다 · 조용한 `pass` 금지"), enum 상수 레지스트리·인터페이스 상수는 Java 에서 일상적 관용구다.
