## 1. R-1 폐쇄 — 픽스처 밖 fresh 실증 (가정 아님)

**보정 내용**: `java_result_from_source()` 가 `SyntaxError` 와 `ImportError`/`OSError` 를
`unavailable_kind: "syntax"` / `"toolchain"` 으로 **구조화 분리**. 문자열 부분일치
`java_analysis_unavailable()` 는 `java_toolchain_unavailable()`·`java_syntax_unavailable()`
두 술어로 대체. coverage skip(`continue`) 은 **toolchain 만** 타고, syntax 는 Python 형제와
같은 **fail-closed 블록**으로 흐른다.

**실증 A/B** — 픽스처 밖 fresh git repo, `@PreAuthorize("hasRole('ADMIN')")` → `USER`
인가 약화 + 유효 `change-intent`:

| 시나리오 | 결과 |
|---|---|
| 파싱 불가 메서드 **덧붙임**(공격) | 🟠 `approval_required` · **exit 2** |
| 미덧붙임(대조) | 🟠 `approval_required` · exit 2 |

**⚠️ 축 분리 주의(1차 시도의 오판을 스스로 교정)**: 최초 exploit repo 를 `app/auth/` 에
두었더니 `**/auth/**` **경로층**(sensitive-zones)이 판정을 대신 실어 rig 를 해도 exit 2 가
유지됐다 — *gov_level 축이 검증되지 않은 가짜 통과*. 경로를 `app/report/`(zone `free`)로
옮겨 **gov_level 단독 축**으로 재현한 뒤에야 유효한 검증이 됐다.

**rig-and-revert (load-bearing 증명)**: `java_toolchain_unavailable()` 을
`in ("toolchain","syntax")` 로 되돌려 **보정 전 동작 재현** →
🟢 **PASS · exit 0 · `reasons: []`** = D-077 이 보고한 원래 구멍 그대로 재현.
복원 → exit 2. ⇒ **신규 가드가 판정을 실제로 지탱한다.**

## 2. R-2 폐쇄 — base 민감 신호 보존

head syntax 실패 시 `sensitive_annotation_records(path, base_result, policy, "base")` 로
base 측 민감 어노테이션을 reason 에 **보존**. 카드 실측:

```
function_protected:app/report/Guard.java::<file>:head:head file could not be parsed
function_protected:app/report/Guard.java::Guard.canTransfer:base:메서드 진입 전 인가 규칙
```

파일이 안 읽혀도 **어떤 함수의 무엇이 위험한지**가 카드에 남는다(승인자 30초 판독 가능).
**rig**: 해당 `records.extend(...)` 한 줄을 `pass` 로 죽이면 base 신호가 **완전 소실**
(grep 0건) — 이 줄도 load-bearing.

  게이트를 직접 로드). ⇒ **거짓 PASS 를 만들 수 없다**(CLAUDE.md 비차단 판정 전 필수질문 통과).

  **후속 가드 권고(TASK-032 AC)**: 명시적 시험 플래그가 켜졌을 때만 override 를 존중하거나,
