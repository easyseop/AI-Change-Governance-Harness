## 🔴 R-3 (보정 사유·신규): 집행-우회 감지가 `continue-on-error: <truthy> # 주석` 인라인 주석 변형을 놓침 → GHA 집행 무력화하며 pass

**근원**: `is_gate_bypass_line` (line 366) — 값 정규화가 unquote·소문자화까지만 하고 **YAML 인라인 주석을 벗기지 않는다.**

```python

if key.strip() != "continue-on-error":
    return False
return unquote_scalar(value).lower() in {"true", "yes", "on"}
```

**fresh 실증(픽스처 밖, `continue-on-error` 라인만 새 추가)**:

| head 삽입 라인 | YAML 파싱값 | 실제 GHA 효과 | 게이트 verdict |
|---|---|---|---|
| `continue-on-error: true # keep going` | `true`(주석 제거) | **집행 무력화** | **pass** ✗ 우회 |

- **YAML 근거(확정적)**: 플레인 스칼라에서 ` #`(공백+`#`)은 인라인 주석 → 파서가 벗겨 값은 `true`. GHA 는 `continue-on-error: true` 로 읽어 스텝 실패를 무시(집행 무력화). 그러나 게이트는 `unquote_scalar(" true # keep going").lower()` = `"true # keep going"` → `{true,yes,on}` 에 없음 → **미탐 → pass**. (`python3` 로 분해 재현 완료.)
- **위반**: AC #7(b) "`continue-on-error` 류 **무력화 삽입**을 감지(결정적 패턴 매칭)". 주석 첨부는 GHA 유효·오히려 **가장 자연스러운 서술 형태**(예: `# ops-approved`)라 R-1 의 대문자/따옴표 변형보다 더 흔하다.
- **§2B 필수 질문("거버넌스 목적에 직접 구멍?") → 그렇다.** R-1 과 **동일 계열**(집행 스위치 OFF·pass = 리뷰어에게 초록불 거짓안심). P0 자기무력화 방지 목적이 직접 뚫림 → **비차단 불가**.
- **무한퇴행 아님(폐쇄 유한·완결)**: GHA/YAML 의 truthy boolean 철자 집합은 **유한·확정**이다. 주석-strip 을 더하면 `{true,yes,on}` × (대소문자·따옴표·공백·인라인주석) 정규화가 GHA 유효 truthy 전부를 덮는다 — R-1 처럼 이 보정으로 계열이 닫힌다(추가 회귀 없음). 이는 D-024 계열의 "지역 dataflow 필요·무한변형" 수용잔여와 성격이 다르다(그건 정적판별 불가·approval 상한이라 catalog 수용; 이건 결정적·완결·P0 집행경로).
- **수정 계약(결정적·최소 델타, `is_gate_bypass_line` 국소)**:
  - `continue-on-error` 값 정규화에 **인라인 주석 제거**를 추가: unquote **안 된** 플레인 스칼라는 첫 ` #`(공백+해시) 이후를 잘라낸 뒤 strip·소문자화해 `{true,yes,on}` 매칭. (따옴표 값은 unquote 후 내부 `#` 이 문자열 일부이므로 별도 처리 불요 — `"true"` 는 이미 `true`.)
