## 4. 🟡 신규 발견 → **차기 AC 가드** (비차단 · TASK-038 에 명시 반영)

> §2B 필수질문 적용 — "이게 거버넌스 목적(민감 변경 포착)에 직접 구멍을 내나?"

> *리그 필요*·*비정상 게이트*·*휴면 정책기능*이다. R-1·R-2 는 **무리깅 기본 경로**에서 재현되던 회귀라 차단했지만
> 이 3건은 그 등급이 아니다. 단 **"관찰"로 흘리지 않고 AC 로 고정**한다.

### G-1 🔴(가장 중요) `try/except` 가드가 **어떤 픽스처에도 고정되지 않음**
`try/except` 만 제거하고 `isinstance` 가드는 남긴 리그로 킷 진입점 전량 실행 → **26/26 PASS**.
즉 신규 회귀 픽스처는 **가드 2개 중 `isinstance` 쪽만 고정**한다.
- 원인: 신규 픽스처의 리그가 `raise RuntimeError("forced evidence failure")` = **V1(파싱되는 형태)** 이라
  `safe_load` 가 예외를 던지지 않는다. V2 경로는 스위트가 한 번도 밟지 않는다.
- 실증: 같은 리그에 V2 입력(`raise RuntimeError("boom: bad value: x")`) → **PyYAML traceback 14줄 재발**.
  ⇒ `try/except` 는 **실제로 load-bearing 인데 테스트가 그걸 증명하지 못한다**.
- Codex 요약의 "각 보정 코드를 임시 제거한 음성검증에서 각각 FAIL" 은 **가드 블록 전체 단위에서는 참**이지만
  **가드 단위로는 과대주장**이다(사실관계만 정정 — 고의성 없음).
- → **차기 AC**: 진입점에 **V2 형태(예외 메시지에 콜론 포함) 케이스 1건 추가** + 음성검증(`try/except` 제거 시 단독 FAIL).

### G-2 게이트 JSON 파싱 실패 시 **2층 블록이 완전 무출력**
`run_gate` 는 `2>&1` 로 stderr 를 합치므로, 게이트가 **stderr 한 줄이라도 뱉고 정상 exit(0/2)** 하면
`json.loads` 가 깨져 `result={}` → `if result:` 가 거짓 → **2층에 헤더만 찍히고 아무것도 안 나온다**(재현 확인).
- 판정은 무손상(cap_exit 는 exit 코드에서 오므로 fail-closed 유지). 현행 게이트는 top-level 이 stdlib 전용이고
  subprocess stderr 를 내부 흡수해 **실측 stderr 0바이트** ⇒ 현재는 휴면.
- 다만 **카드 파싱 실패에는 "주입 불가" 고지를 넣었으면서 게이트 출력 파싱 실패에는 아무 고지가 없다**(비대칭).
  tree-sitter 등 동적 로드(`importlib.util.exec_module`)가 경고를 흘리면 밟힌다.
- → **차기 AC**: `if result:` 에 `else` 고지 1줄(`능력 게이트 출력 파싱 불가 — 원문 앞 N줄`) + 회귀 케이스.

### G-3 shadow 렌더러가 `level=` 을 잃음
`main`: `shadow: app/service.py::outbound_network level=protected` ↔ 브랜치: `shadow: app/service.py::outbound_network`(재현 확인).
성숙도 shadow 는 비집행이라 판정 영향 0 이고 **동봉 카탈로그에 `maturity: shadow` 항목이 없어 현재 휴면**이지만,
승격 시 영향도를 읽는 정보가 빠진다. → **차기 AC**: `level=` 복원(1줄) + shadow 픽스처.
