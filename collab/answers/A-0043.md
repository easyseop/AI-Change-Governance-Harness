## 1. 결론

## 2. ✅ R-1 해소 확인 (실증)

### 2.1 보정 내용

`method_targets()` 에서 두 블록이 삭제됐다 — `if interface_receiver and implementations…: owners = set()`
(A-0042 O-1 이 지적한 **무효 코드**)와 `owners.difference_update(interface_names)`(**결함의 근원**).
남은 본문은 `type_names_by_simple` ∪ `implementations` ∪ `ancestors` 로, 인터페이스 소유자를 빼지 않는다.

### 2.2 A-0042 §2.2 적대입력 4종 — 픽스처 밖에서 **새로 작성**해 재실행

| # | 입력 | A-0042 (보정 전) | 현재 (보정 후) |
|---|---|---|---|
| A | `default`·repo 내 구현 **0개** | 엣지 없음 · `unresolved` | ✅ `Caller.run → P.audit` |
| B | 구현체가 오버라이드 | ✅ `Impl.audit` 만 | ✅ `Impl.audit` **+ `P.audit`**(superset 강화) |
| C | 인터페이스 `static` | 엣지 없음 · `unresolved` | ✅ `Caller.run → P.util` |
| D | 인터페이스 `private` 헬퍼 ← `default` 본문 자기호출 | 엣지 없음 · `unresolved` | ✅ `Caller.go → P.d` · `P.d → P.helper` |

4종 모두 `unresolved_calls` **비었고** `errors` 비었다 — 즉 미해소로 새는 게 아니라 정식 엣지가 됐다.

### 2.3 하류 end-to-end — verdict 가 실제로 뒤집힌다

`check-indirect-impact.py` 를 직접 import 해 그 `adjacency_from_edges`·`reachable_paths` 로 확인:

```
H  interface default body [R-1 시나리오]
   edges : AuditPort.audit→AuditPort.settle · Caller.run→AuditPort.audit
   sink=Caller.run (forward 2hop) → 변경함수 'AuditPort.settle' 탐지? True
   path  = ['Caller.run','AuditPort.audit','AuditPort.settle']   ⇒ approval_required
```
A·C·D 도 각각 `approval_required`. **보정 전 `pass` → 보정 후 `approval_required`** = 탐지 구멍 폐쇄.

> **자기정정(A-0042 §2.3 방향 표기 오류)**: `reachable_paths` 는 sink 에서 `caller→callee` 방향으로
> **전진** 탐색한다(TASK-037 AC#3 "sink 로부터 forward N홉"). A-0042 에서 나는 이를 "역도달성"으로
> 표기했다 — **방향 라벨이 틀렸다.** 다만 당시 해당 repo 는 엣지가 **literally NONE** 이라
> 어느 방향으로도 도달 불가였으므로 **결론(미탐지·조용한 pass)은 그대로 유효**하다. 표기만 정정한다.

### 2.4 rig-and-revert — 보정이 load-bearing 인가

**RIG-1** 보정 전 제외 로직(두 블록 모두) 복원 → `java-callgraph-conservative` **단독 FAIL**,
**142/144 · adversarial 7/8**. Codex 가 handoff 에 적은 수치와 **정확히 일치**(사실 확인).
⇒ 신규 기대값이 보정을 실제로 붙잡고 있다. ✅

## 3. 🔴 R-2 (신규 · 선재) — `interface … extends …` 미인식 ⇒ 서브인터페이스 계층 엣지 소실

### 3.1 위치·원인

`.harness/gates/extract-java-callgraph.py` 88–93:

```python
def declaration_supertypes(source_bytes, node):
    names = []
    for field in ("superclass", "interfaces"):        # ← 필드 2개만 본다
        names.extend(type_names(source_bytes, node.child_by_field_name(field)))
    return sorted(set(names))
```

tree-sitter-java 를 직접 덤프해 확인한 결과:

| 선언 | 자식 노드 | `superclass` 필드 | `interfaces` 필드 | `declaration_supertypes` |
|---|---|---|---|---|
| `class C implements B` | `super_interfaces` | None | **`super_interfaces`** | `['B']` ✅ |
| `interface B extends A` | **`extends_interfaces`** | None | **None** | **`[]`** ❌ |

`interface X extends Y` 의 상위타입은 `extends_interfaces` **자식 노드**로 오고, 조회하는 두 필드명 중
**어느 것으로도 노출되지 않는다.** ⇒ 인터페이스→인터페이스 상속이 `implementations`·`ancestors` 양쪽에서
**완전히 비가시**. 픽스처 실제 맵으로 확인:

```
implementations: {'CardPayment': ['RegionalCardPayment'],
                  'PaymentPort':  ['CardPayment','RegionalCardPayment','WirePayment']}
```
— 인터페이스 간 링크가 **한 건도 없다.**

### 3.2 fresh 적대입력 — 실제 런타임 엣지 소실 (구조 동형 대조군은 정상)

Spring 계층(`interface Repo extends BaseRepo`)에서 표준인 모양:

```java
interface BasePort { void settle(); }
interface PaymentPort extends BasePort { }
class PaymentImpl implements PaymentPort { public void settle(){ } }
class Caller { BasePort base; void go(){ base.settle(); } }
```

| 케이스 | 산출 엣지 | 실제 런타임 타깃 |
|---|---|---|
| **I** 서브인터페이스 `extends` | `Caller.go → BasePort.settle` **뿐** | `PaymentImpl.settle` ← **소실** |
| **J** 동일 구조를 추상클래스로 | `Caller.go → BasePort.settle` **+ `PaymentImpl.settle`** ✅ | — |

`interface`↔`abstract class` 한 단어만 바꾸면 결과가 갈린다 ⇒ **의도상 버그 확정**(R-1 과 동일 논법).
`interface B extends A` 의 `default` 오버라이드도 같은 이유로 소실된다(`Caller.go → B.f` 없음).

### 3.3 §2B 필수질문 = **예** (직접 구멍) — end-to-end 실증

```
I  서브인터페이스 extends 체인
   sink=Caller.go (forward 2hop) → 변경함수 'PaymentImpl.settle' 탐지? False   ⇒ verdict: pass
J  추상클래스 대조군
   sink=Caller.go (forward 2hop) → 변경함수 'PaymentImpl.settle' 탐지? True    ⇒ verdict: approval_required
```

두 케이스 모두 `coverage.unevaluated` **빈 배열**, `errors` **빈 배열**이다.
⇒ A-0042 §2.3 에서 확립한 그대로 — `unevaluated` 는 verdict 무기여, fail-closed 트리거는 `errors` 뿐 —
**하류에서 구제되지 않고 조용한 `verdict: pass`** 가 된다. 게다가 미해소 목록에조차 안 남으므로
**"놓쳤다는 사실 자체가 어디에도 안 보인다"** — R-1 보다 오히려 은닉성이 높다.

AC#2 ① 은 "**인터페이스 메서드 호출 → repo 내 그 인터페이스의 모든 구현체(`implements`/`extends` 열거)로 엣지**"
라고 명시한다. `extends` 열거가 없으므로 **수용기준 미충족**이다(형식 미달이 아니라 실효 미달).

### 3.4 보정 방향 — 3줄 · 회귀 0 (내가 직접 패치해 확인)

```python
def declaration_supertypes(source_bytes, node):
    names = []
    for field in ("superclass", "interfaces"):
        names.extend(type_names(source_bytes, node.child_by_field_name(field)))
    for child in node.children:                                   # ← 추가
        if child.type in {"extends_interfaces", "super_interfaces", "superclass"}:
            names.extend(type_names(source_bytes, child))
    return sorted(set(names))
```

- 케이스 **I** → `Caller.go → PaymentImpl.settle` **복원** ✅ / 케이스 **F**(`B extends A` + `default` 오버라이드) → `Caller.go → B.f` 복원 ✅
- **전체 스위트 143/144 유지 = 회귀 0** (실패는 기존 `tree-sitter-smoke` 환경 건뿐)
- 구현은 Codex 재량 — 위는 "고칠 수 있고 값이 싸다"의 실증이지 지시가 아니다.

### 3.5 AC#5 고정 적대 세트에 추가 요청 (+ 음성검증)

현재 픽스처에는 **인터페이스가 인터페이스를 상속하는 케이스가 한 건도 없다** ⇒ 무보호.
- ① `interface B extends A` + 구현체는 `B` 만 선언 → `A` 타입 수신자 호출이 구현체까지 엣지
- ② `B` 가 `A` 의 `default` 를 오버라이드 → 두 소유자 모두 타깃(superset)
- ③ **음성검증**: `extends_interfaces` 처리를 되살린 뒤 제거하면 해당 케이스가 **단독 FAIL**
- ④ **추상클래스 대조군**(§3.2 J)도 함께 픽스처화 — A-0042 O-2 가 지적한 조상확장 무보호도 같이 닫힌다
