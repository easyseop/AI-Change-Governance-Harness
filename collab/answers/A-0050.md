## 3. 🔴 O-20 (신규 · AC#3 이 만든 오탐) — `has_static_modifier` 과대매칭

```python
return any("static" in re.findall(r"[A-Za-z_][A-Za-z0-9_]*", node_text(source_bytes, child))
           for child in node.children)
```

`node.children` 은 `modifiers` 뿐 아니라 **타입·declarator(초기화식 포함) 전부**다. 따라서 **초기화식 텍스트에 `static` 토큰이 들어가면 instance 필드가 `<clinit>` 로 오표기**된다.

**실증(fresh)**:
```java
class Flow { void log(String v){int m=1;} Task task = () -> log("static"); }
```
- `main`: `caller: Flow.<init>` ✅ (정확)
- 브랜치: `caller: Flow.<clinit>` ❌

원 결함(`static final`·`private static`·어노테이션 동반)은 닫혔으나(RIG-A 단독 FAIL) **반대편 오표기를 새로 만들었다** — 감사카드가 인스턴스 생성 시점 코드를 클래스 로드 시점으로 오도. **판정 무영향**(초기화 레코드는 dispatch 축으로 판정되고 합성 id 는 그래프 노드가 아님)이므로 **차단은 아니나**, AC#3 이 고치려던 결함과 **동종·동급**이므로 이번 보정에서 함께 닫는다.
**수정**: `child.type == "modifiers"` 인 자식으로 한정한 뒤 토큰 분해.

## 4. 🔴 자기정정 (4회 연속 — 지시 결함)

**AC#1 이 "N4 를 `pass` 단언 케이스로 신설" 을 불변식으로 못박은 것이 이번 결함의 근본원인이다.**

- §1.4 가 보여주듯 **N4-as-`pass` 는 건전한 판별식과 양립하지 않는다.** N4 의 sink 막다른 길은 `Flow.check` 의 `rows.size()`(외부 호출)이고, 그 외부 호출이 콜백을 되부를 수 있는지는 **정적으로 알 수 없다**. dispatch 대상이 **불명**인 지연 콜백은 보수적으로 "sink 막다른 길에서 호출될 수 있다" 고 봐야 한다.
- 게다가 N4 픽스처는 변경 함수에 **`Flow.log`(= 지연 람다 본문이 호출하는 함수)** 를 포함한다 ⇒ "변경이 지연 본문에 닿는가" 같은 대안 축으로도 `pass` 가 나오지 않는다.
- ⇒ **N4 를 `pass` 로 만들 수 있는 축은 사실상 `caller` 축뿐이었고, 그 축은 A-0049 자신이 틀렸다고 배제한 축이다.** AC#1 은 **서로 모순된 두 요구**(caller 축 금지 + N4 pass)를 동시에 걸었다.
- **Codex 는 두 요구를 모두 만족시키는 유일한 형태를 구현했다 — §1 위반도 구현 품질 문제도 아니다.** A-0046→A-0047(O-10), A-0047→A-0048(R-1), A-0048→A-0049(O-14) 에 이어 **4회째 같은 유형의 지시 결함**이다.

**O-14(소음)의 현재 위치**: `caller` 축을 제거하면 N4 는 다시 `approval_required` 로 돌아간다 = **O-14 는 이 보정으로 닫히지 않는다.** 이는 **의도된 후퇴**다 — 과소탐과 과탐 중 과탐을 택하는 것이 MVP-3 계약이다. O-14 를 건전하게 닫으려면 **막다른 길 쪽을 좁혀야 한다**(예: 미해소 호출의 **수신자 선언 타입**을 기록해 지연 레코드의 **대상 타입**과 일치할 때만 막다른 길로 계수). **이 방향은 미실증이므로 이번 보정에 요구하지 않는다** — TASK-041 로 분리하고, 착수 시 §4 의 실증 절차를 그대로 밟는다.
