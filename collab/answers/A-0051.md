## 4. 🔴 R-3 (차단) — 해소된 dispatch 대상 + **불투명 sink 호출지점** ⇒ 과소탐

### 4.1 논리

교집합 축은 **양쪽이 다 해소될 때만** 건전하다. sink 쪽 호출지점이 **미해소**면 막다른 길은 `Task.exec`(대상) 이 아니라 **`Flow.sink`(미해소 호출의 caller)** 로 기록된다 ⇒ 교집합이 **구조적으로 항상 공집합** ⇒ 조용히 드롭.

즉 배제가 발동하는 이유가 **"타입이 실제로 무관해서"(F6 = 건전)** 가 아니라 **"분석이 sink 쪽을 못 풀어서"(불건전)** 인 경우가 통째로 열려 있다.

### 4.2 실증 — 픽스처 밖 fresh, `main`↔브랜치 동일 rev-range

전부 별도 임시 repo · sink `Flow.sink` · 변경 = `Vault.transfer` 본문 1→2.

| # | 형태 | `main` | 브랜치 |
|---|---|---|---|
| **G1** | `Task task = () -> new Vault().transfer();` + `List<Task> tasks;` + `sink(){ tasks.get(0).exec(); }` | `approval_required`(2) | **`pass`(0)** |
| **G2** | 같은 필드 + `Task get(){return task;}` + `sink(){ get().exec(); }` | `approval_required` | **`pass`** |
| **H1** | `mid(){ task.exec(); }` + `sink(){ mid(); rows.size(); }`, **`hops: 1`** | `approval_required` | **`pass`** |
| G3 대조군 | `sink(Task t){ t.exec(); }` (파라미터 타입 해소) | `approval_required` | `approval_required` ✅ |
| G4 대조군 | `sink(){ Task t = task; t.exec(); }` (지역변수 해소) | `approval_required` | `approval_required` ✅ |

G1 coverage(브랜치) — **정직히 찍히고 판정만 조용히 통과**:

`sink_dead_ends = {Flow.sink}` ↔ `dispatch_targets = {Task.exec}` → 교집합 ∅ → 드롭. `fail_closed: []`.

**H1 은 `hops` 경계축**(A-0049 §3 검증축 ⑦)이다 — 대상이 폐쇄 안에 있어도 **hops 밖이면** 같은 방식으로 드롭된다.

### 4.3 왜 차단인가 (§2B 필수질문 = **예**)

- `tasks.get(k).exec()` · `get().exec()` · `registry.find(k).handle()` 은 **콜백 레지스트리/디스패처의 표준형**이다. 수신자가 컬렉션·게터·캐스트에서 오는 순간 sink 쪽은 항상 미해소다.
- 방향이 **과소탐**이고 **`main` 보다 나쁘다** ⇒ MVP-3 "과탐 반올림 · **과소탐 금지**" 정면 위반. **비차단 금지.**

## 5. 🔴 R-4 (차단) — 인자전달 람다가 **해소되면 fail-closed 표식이 통째로 사라진다**

### 5.1 논리 — 두 경로의 비대칭

`extract-java-callgraph.py` `collect_calls()`:
- **필드 초기화 경로**(`scan_deferred`): dispatch 대상이 해소돼도 **deferred 레코드를 남긴다**(G1 이 그 모양).
- **메서드 본문 경로**(`:700-707`): 대상이 해소되면 **엣지만 만들고 `unresolved.add(...)` 를 하지 않는다** — `else` 갈래에서만 레코드가 생긴다.

⇒ 후자는 sink 가 그 인터페이스를 **불투명하게** 부르면 **엣지도 안 닿고 deferred 레코드도 없어** `java_deferred` 가 비고, **fail-closed 가 아예 발동하지 않는다.**

### 5.2 실증 — G6(레지스트리 표준형, fresh)

```java
interface Task { void exec(); }
class Vault { void transfer() { int value = 1; } }              // 변경 대상
class Registry { Map<String,Task> m = new HashMap<>();
                 void put(String k, Task t){ m.put(k,t); }
                 Task find(String k){ return m.get(k); } }
class Flow { Registry reg = new Registry();
             void wire(){ reg.put("pay", () -> new Vault().transfer()); }
             void sink(String k){ reg.find(k).exec(); } }
```
- `main`: **`approval_required`(2)**
- 브랜치: **`pass`(0)** — `coverage.unevaluated` 에 `lambda_dispatch` 레코드가 **0건**, `fail_closed: []`, `errors: []` = **완전히 깨끗한 통과**

R-3 보다 나쁘다: R-3 은 최소한 coverage 에 흔적이라도 남지만, **R-4 는 흔적조차 없다.** 이 상태에서 `Task.exec → Vault.transfer` 엣지는 **존재하는데** `Flow.sink → Task.exec` 만 없어서 경로가 끊긴다.

**책임 소재**: `invocation_parameter_type`(1차 제출분)은 A-0050 §5 에서 **RIG-D 로 load-bearing 확인**하고 통과시킨 기능이다. 그때 **sink 쪽이 불투명한 경우를 대조하지 않았다** → §7 자기정정.

## 7. 🔴 자기정정 (5회 연속 — 지시·검증 결함)

- ⇒ **교훈(이번에 기록)**: **판별식을 좁히는 변경은 좁힘의 *반대 조건*(여집합)을 fresh 입력으로 반드시 쳐본다.** rig-and-revert 는 *추가된* 단언만 검증하고 *사라진* 커버리지는 검증하지 못한다. A-0047→A-0051 5회 모두 이 자리에서 났다.
