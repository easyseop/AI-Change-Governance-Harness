## 1. 결론

핵심 골격은 옳다. 결정론·오버로드 합집합 병합·어노테이션 라인 포함·파싱실패 fail-closed·
Python 추출기 무개조·스코프 청결까지 전부 확인했고, 팬아웃 로직은 rig-and-revert 로 load-bearing 임을 실증했다(§4).

그러나 **AC#2 의 핵심 계약인 "superset(진짜 엣지 안 놓침)"이 깨지는 경로를 찾았다**:
**인터페이스가 직접 들고 있는 메서드 본문(`default`·`static`·`private` 및 인터페이스 내부 자기호출)은
어떤 경우에도 엣지 대상이 되지 못하고 통째로 `unresolved_calls` 로 흘러간다.**

이건 "미해소 정직 노출"(AC#3)이 아니라 **정적으로 완전히 해소 가능한 호출을 잃는 것**이고,
`coverage.unevaluated` 는 판정에 **아무 영향을 주지 않는 순수 보고 필드**라 하류에서 구제되지 않는다(§2.3).
결과적으로 TASK-037 이 이 그래프를 소비하면 **sink 상류 변경이 `pass` 로 통과한다** — 탐지 구멍.

## 2. 🔴 R-1 — 인터페이스 선언 본문이 엣지 대상에서 영구 제외됨 (AC#2 superset 위반)

### 2.1 위치

`.harness/gates/extract-java-callgraph.py` `method_targets()` 285–293:

```python
owners = set(type_names_by_simple.get(owner_type, set()))
interface_receiver = any(owner in interface_names for owner in owners)
if interface_receiver and implementations.get(owner_type):
    owners = set()                       # ← (b) 사실상 무효 코드, §3 O-1
owners.update(implementations.get(owner_type, set()))
for owner in list(owners):
    owners.update(ancestors.get(owner, set()))
if interface_receiver:
    owners.difference_update(interface_names)   # ← (a) 결함의 근원: 무조건 제외
```

수신자가 인터페이스면 **인터페이스 자신을 타깃 소유자에서 무조건 뺀다.** 순수 추상 메서드라면 맞지만,
Java 8+ 인터페이스는 **본문 있는 메서드를 직접 소유**한다. 그 본문이 타깃이 될 길이 사라진다.

### 2.2 fresh 적대 입력 4종 — 전부 실제 엣지 소실 (픽스처 밖에서 직접 작성·실행)

| # | 입력 | 실제 런타임 엣지 | 게이트 출력 |
|---|---|---|---|
| A | `interface P { default void audit(){} }` · repo 내 구현 **0개** · `p.audit()` | `C.run → P.audit` | **edges 없음** · `unresolved: p.audit` |
| B | 구현체가 `audit()` 를 **오버라이드** | `C.run → Impl.audit` | ✅ `C.run → Impl.audit` — **유일하게 동작하는 경우** |
| C | `interface P { static void util(){} }` · `P.util()` | `C.run → P.util` | **edges 없음** · `unresolved: P.util` |
| D | `interface P { private void helper(){} default void d(){ helper(); } }` | `P.d → P.helper` | **edges 없음** · `unresolved: helper` |

즉 **구현체가 오버라이드한 경우(B)만 동작**하고, 인터페이스 본문이 실제 타깃인 A·C·D 는 전부 소실된다.
D 는 특히 넓다 — **인터페이스 `default`/`static` 본문 안의 무수식 자기호출은, 그 인터페이스에
구현체가 하나라도 있으면 전부 소실**된다(수신자 = 자기 자신 = 인터페이스 → 같은 경로).

**구조 동형인 추상클래스 대조군은 정상 동작**하므로 설계 의도상 버그임이 분명하다:

```java
abstract class Base { void template(){ step(); } abstract void step(); }
class Impl extends Base { void step(){} }
class Runner { Base b; void go(){ b.template(); } }
```
→ `Base.template→Base.step` · `Base.template→Impl.step` · `Runner.go→Base.template` **전부 정상**.
같은 구조를 `interface` 로만 바꾸면 **엣지 0개**가 된다.

### 2.3 하류 실증 — `coverage.unevaluated` 는 구제해주지 않는다 (탐지 구멍 확정)

`check-indirect-impact.py` 를 실제로 import 해서 그 `adjacency_from_edges`·`reachable_paths` 로 돌렸다.
입력: sink = `AuditPort.settle`, 변경함수 = `Caller.run`, 실제 경로 = `Caller.run → AuditPort.audit → AuditPort.settle`(2홉).

```
edges       : NONE
unevaluated : [('AuditPort.audit','settle'), ('Caller.run','port.audit')]
reverse-reachable from sink @1/2/3 hop : NONE / NONE / NONE
=> 'Caller.run' 이 sink 상류로 탐지되는가? False
```

그리고 `check-indirect-impact.py` 를 한 줄씩 확인한 결과 — **`coverage.unevaluated` 는 254–256 줄에서
그대로 통과 출력될 뿐 `fail_closed_records` 에도 verdict 에도 기여하지 않는다.**
fail-closed 를 트리거하는 건 `callgraph["errors"]`(197–199) 뿐이고, 미해소 호출은 `errors` 가 아니다.
verdict 는 `indirect_impact or fail_closed_records` 로만 결정된다(244–248).

⇒ **엣지가 빠지면 그냥 `verdict: pass`.** "정직하게 노출했으니 괜찮다"가 성립하지 않는 구조다.
AC#3 의 `unresolved_calls` 는 **리플렉션처럼 원리적으로 못 푸는 것**을 위한 자리이지,
정적으로 풀 수 있는 호출의 도피처가 아니다. 지금은 미해소 목록을 **거짓으로 부풀리는** 부작용도 있다.

### 2.4 픽스처가 결함을 *고정*하고 있다 (음성검증에서 드러남)

`java-callgraph-conservative` 의 기대 엣지 집합에는 `PaymentPort.pay` 가 없다. 그래서
**(a) `owners.difference_update(interface_names)` 를 제거하면 이 케이스가 FAIL 한다**(143/144 → 142/144).
즉 현재 테스트는 이 제외 동작을 **정답으로 못박아** 두었다. 보정 시 픽스처 기대값도 함께 고쳐야 한다.

### 2.5 보정 방향 (제안 — 구현은 Codex 재량)

- `owners.difference_update(interface_names)` 를 **무조건 제외 → 조건부**로: 인터페이스가 **본문을 가진**
  메서드(`default`/`static`/`private`, 즉 `method_declaration` 에 `body` 존재)를 소유하면 그 소유자는 남긴다.
- 더 단순하고 superset 방향으로 확실한 대안: **인터페이스 소유자를 아예 빼지 않는다.**
  추상 선언까지 타깃에 들어와 엣지가 늘지만 그게 정확히 AC#2 가 요구하는 과대근사 방향이고,
  인터페이스 **선언 자체의 변경**도 sink 상류로 잡히는 부수 이득이 있다. (§3 O-1 의 무효 코드도 같이 정리됨)
- **AC#5 고정 세트에 추가 + 음성검증**: ①`default` 메서드(미오버라이드·구현 0개 둘 다) ②인터페이스 `static`
  ③인터페이스 본문 내부 자기호출 — 각각 엣지 존재를 단언하고, 위 제외 로직을 되살리면 그 케이스가 **단독 FAIL** 할 것.
- 회귀 방지로 **§2.2 의 추상클래스 대조군**도 픽스처화 권장(현재 무보호 — O-2).

## 3. 비차단 관찰 (보정 시 같이 보면 좋음 · 단독 반려 사유 아님)

- **O-1 무효 코드**: `if interface_receiver and implementations.get(owner_type): owners = set()` (287–288) 는
  **제거해도 스위트 143/144 무변화**. 뒤의 `difference_update` 가 같은 일을 이미 하기 때문에 도달 효과가 없다. R-1 보정 시 함께 정리.
- **O-2 조상(ancestors) 확장이 무보호**: 290–291 을 제거해도 **143/144 그대로 PASS**. 로직은 정확하지만
  (내가 fresh 입력으로 동작 확인) **어떤 픽스처도 짚지 않아** 조용히 퇴화할 수 있다. 픽스처 1건 권장.
- **O-3 픽스처의 `@Autowired`/`@Transactional` 이 무기능(사실 정정)**: 두 어노테이션을 픽스처에서 **삭제해도
  케이스는 그대로 PASS**(143/144). 구현은 어노테이션을 전혀 읽지 않고 **순전히 타입 기반**으로 해소한다.
  결과적으로 AC#2 ②③ 의 *효과*는 충족되지만(오히려 더 일반적), handoff·summaries 의
  "`@Autowired` 필드·생성자 주입을 모든 구현체로 fan-out … `@Transactional` 호출은 직접 엣지로 유지" 표현은
  **어노테이션 인식 로직이 있는 것처럼 읽힌다 — 과대 서술**. 또 AC#1 의 "TASK-031 어노테이션 재사용"은 실제로는 안 했다
  (필요가 없어서 — 타당하지만 명시 필요). 문구 정정 권장.
- **O-4 익명 내부클래스의 유령 노드**: `new Runnable(){ public void run(){…} }` 가 바깥 메서드와 같은 소유자로
  기록돼 `Svc.run` 노드가 **2개**(line 3, line 4) 생기고 실제 `Svc.run` 과 id 가 충돌·병합된다.
  엣지가 늘어나는 **과대근사 방향이라 안전**하지만 노드 목록이 오염된다.
- **O-5 파서 무캐시**: `parse_source` 가 파일마다 `make_java_parser()` 를 호출해 `tree_sitter_java`
  import + `Language`/`Parser` 생성을 **파일 수만큼 반복**. 모듈 레벨 1회 또는 `lru_cache` 로 hoist 권장.
- **O-6 제네릭 타입 선택이 자의적**: `declared_type` 이 `type_names`(**알파벳 정렬된 set**)의 `[-1]` 을 쓴다.
  `Map<PaymentPort, String> m` → `String` 으로 해소되어 `PaymentPort` 를 잃는다. 결정론은 유지되나 근사 방향이 입력에 따라 뒤집힌다.
- **O-7 parity 픽스처가 최소치**: `sink→helper` 1홉·다형성 없음이라 **TASK-036 의 핵심인 과대근사를 전혀 검증하지 않는다.**
  (케이스를 언어별로 따로 두는 형식 자체는 기존 `parity-java-capability-*` 선례와 일치 — 형식은 문제 없음.)
  인터페이스 다형성 ↔ Python 덕타이핑 대응 쌍이었다면 R-1 이 여기서 잡혔을 것. TASK-037 AC#5 에서 보강 권장.
