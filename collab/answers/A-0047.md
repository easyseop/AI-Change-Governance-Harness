## 4. 신규/이월 관찰 — **🔴 차기 AC(TASK-039)로 명시 고정**

§2B 필수질문("거버넌스 목적에 직접 구멍을 내나?")에 **예**인 항목이 둘 있다. 규정대로 비차단으로 흘리지 않고 **차기 AC 가드**로 막는다. **반려하지 않는 이유**: 둘 다 **이번 델타가 만든 회귀가 아니고**(보정 전·`main` 대비 후퇴 없음), 하나는 A-0045·A-0046 두 번의 리뷰에서 내가 지명하지 못한 **선재 결함**이며(뒤늦은 골포스트 이동 금지), Java L3 부재 상태(`main`)보다 명백히 낫기 때문이다.

- **🔴 O-9 (신규 · 선재) 메서드 본문 *밖*의 dispatch 는 완전 무표식**
  `collect_calls` 는 `methods`(= `method_declaration`·`constructor_declaration`) 본문만 순회한다 ⇒ **필드 이니셜라이저·static/instance 이니셜라이저 블록**의 호출은 **엣지도 `coverage` 기록도 없다**. fresh 실증(보정 전·후 **동일**, 즉 선재):
  | 입력 | 결과 |
  |---|---|
  | `Task task = () -> new Vault().transfer();` (**필드 초기화 람다**) | **`pass`(0) · coverage 빔** |
  | `Task task = new Task(){ public void exec(){ new Vault().transfer(); } };` (**필드 초기화 익명** = AC#6 이 지명한 바로 그 모양) | **`pass`(0) · coverage 빔** |
  | `static { task = () -> new Vault().transfer(); }` | **`pass`(0) · coverage 빔** |
  | `Gateway() { task = () -> new Vault().transfer(); }` (**생성자**) | `approval_required`(2) · 실 발견 ✅ |
  `private final Handler h = () -> …` 는 Java DI/레지스트리 배선의 최빈형이다. **AC#6 의 문언("조용한 `pass` 금지")은 위치를 한정하지 않으므로 범위상 미완**이다.
- **🔴 O-10 (신규) 승격이 repo 전역·무조건 ⇒ 과탐 소음이 층을 무력화할 수 있음**
  승격 필터는 `java_graph.coverage.unevaluated` **전량**을 본다 — sink 와의 관계도, sink 존재 여부도 보지 않는다. fresh 실증: **등록 sink 가 0개인 repo**에서 `rows.forEach(r -> log(r))` 한 줄만 있어도 무관 파일 변경이 **`approval_required`·`tool_owner`**(보정 전 `pass`). 실서비스 Java repo 라면 **거의 모든 Java PR 이 상시 승인요구** = 경보 피로 → 진짜 🔴 가 묻힌다.

  **내가 짠다면**: 승격을 *관련성*으로 게이팅한다 — **sink 전방폐쇄(N홉) 안에 "본문 없는 추상/인터페이스 메서드 = 막다른 길" 이 존재할 때만** 미해소 dispatch 를 승격. 이번 fresh 세트에서 이 판별식은 **ADV-1·3·6·7·11·12 는 전부 승격 유지, ADV-8·13(소음)만 `pass`** 로 정확히 갈린다(수기 대조 확인). 과탐 방향은 유지하면서 무관 소음만 제거된다.
