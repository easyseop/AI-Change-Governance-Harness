## 2. 질문 3 — 스키마 계약 (제안 대비 **4곳 수정**)

제안: `annotations.<SimpleName> = {level, entrypoint, source, owner}`.
방향은 맞다. 다만 그대로 두면 **과소탐 구멍 2개 + 하류 정보부족 2개**가 생긴다. 확정 계약은 아래.

```yaml
annotations:                       # ← map 이 아니라 **list of records** (형제 정책 sensitive-capabilities 와 동형)
  - name: PreAuthorize             # simple name
    level: protected               # protected | watched  (frozen 금지)
    entrypoint: false
    reason: <카드에 그대로 실릴 한 줄>
    reviewer: security-reviewer    # approval-routing.yaml 어휘
    fqns: [org.springframework...] # 감사·문서용. **매칭에 쓰지 않는다**
    when: {argument: nativeQuery, equals: "true"}   # 선택 — 인자 조건부 항목만
    source: {framework: spring-security, builtin: cwe}
    owner: security-reviewer
```

**수정 ①(🔴 과소탐 — 반드시 반영) 매칭은 "마지막 점-세그먼트" 기준.**
Java 는 `@PreAuthorize` 와 `@org.springframework.security.access.prepost.PreAuthorize` 를
**둘 다 허용**한다. 어노테이션 원문 토큰을 simple name 과 그대로 비교하면 FQN 인라인 표기를
**놓친다**. 놓침 = MVP-3 공통 "과소탐 금지" 정면 위반이다. 반드시 마지막 세그먼트로 정규화해 대조하라.
`fqns` 는 매칭용이 아니다(임포트 해소는 이 층 범위 밖 — 그래서 simple name 대조가 과탐 방향으로 안전하다).

**수정 ②(🔴 과소탐 — 반드시 반영) 인자 조건 `when` 이 스키마에 필요하다.**
AC#2 는 `@Query(nativeQuery=true)` → protected 인데, **JPQL `@Query` 는 신호가 아니다.**
제안 계약에는 인자를 볼 자리가 없어 둘 중 하나가 된다 — 전부 protected(과탐 폭주로 신호 소멸) 또는
전부 무시(원시 SQL 표면 **놓침**). 그래서 `when: {argument, equals}` 를 넣었다.
**미해소 시 태도는 정책이 정한다**: `defaults.unresolved_argument: match` —
`@Query(nativeQuery = SOME_CONSTANT)` 처럼 상수 참조라 결정적으로 못 풀면 **추정하지 말고 매칭으로 취급**
(= protected). 값 추정 금지 + 과탐 반올림 원칙 동시 충족.

**수정 ③ map → list.** 형제 정책(`sensitive-capabilities.yaml`)과 동형이고, **중복 `name` 을
검증오류로 잡을 수 있다**. map 이면 중복 키가 조용히 마지막 승리로 덮여 등급이 소리 없이 바뀐다.
**중복 name → 검증오류(fail-closed)**.

**수정 ④ `reason` · `reviewer` 필수.** 제안에 없었다. `reason` 없이는 카드가 "PreAuthorize 변경됨"
까지만 말하고 사람이 30초 안에 위험을 파악할 문장이 안 나온다(CLAUDE.md §2C). `reviewer` 없이는
approval-routing 이 누구에게 보낼지 못 정한다.

## 3. 질문 2 — source / owner 확정값

`source` 는 형제 정책의 `builtin:` / `org:` 에 **`framework:` 를 추가**해 3종으로 쓴다
(프레임워크 공식 의미론이 근거인 항목이 다수라 `builtin`·`org` 어느 쪽으로도 정직하지 않다).
확정값은 정책 파일에 전부 기입했다. 요약:

| 항목 | level | entrypoint | source | owner/reviewer |
|---|---|---|---|---|
| `@PreAuthorize` `@PostAuthorize` `@Secured` `@RolesAllowed` `@PermitAll` `@DenyAll` | **protected** | false | framework: spring-security / jakarta-annotations + builtin: cwe(285·306) | security-reviewer |
| `@Query(nativeQuery=true)` `@Modifying` | **protected** | false | framework: spring-data-jpa (+cwe-89) | dba-reviewer |
| `@Transactional` | watched | false | framework: spring-tx + org: governance-wg | dba-reviewer |
| `@RequestMapping` `@Get/Post/Put/Patch/DeleteMapping` | watched | **true** | framework: spring-web | api-owner |
| `@Scheduled` `@EventListener` `@KafkaListener` | watched | **true** | framework: spring-context / spring-kafka | platform-reviewer |

**AC#2 대비 추가분 3종**(정책 소유자 판단 — 전부 과탐 방향):
`@PermitAll`·`@DenyAll`(인가 **해제/차단** 선언 = 인가 약화의 가장 직접적 신호. 형제 4종만 넣고
이 둘을 빼면 "인가 약화 포착"에 구멍이 남는다) · `@Put/Patch/DeleteMapping`
(AC 는 Get/Post/RequestMapping 만 예시했으나 DELETE 진입점을 빼는 건 정당화가 안 된다).

## 4. 🔴 불변식 3개 — 구현이 반드시 지킬 것

**불변식 1 — 이 카탈로그의 level 은 `protected|watched` 뿐. `frozen` 금지.**
`@Transactional` 을 붙인 사람이 "이 함수를 얼려달라"고 선언한 적은 없다. 프레임워크 어노테이션은
**추론 신호(2·3층)** 이고 2·3층은 **자동 차단 금지**(CLAUDE.md §4 · D-004).
정책에 `frozen` 이 오면 → **검증오류 기록 + protected clamp**(조용한 무시 금지).
↔ **대비**: `@Gov(level=frozen)` 은 **저자의 명시 선언** = 1층 등가 → `blocked` 가능(AC#1, Python `@gov` parity).
⇒ 게이트는 **신호 출처(declared `@Gov` vs inferred framework)를 끝까지 구분해 보존**해야 한다.
합쳐서 max 만 취하면 프레임워크 카탈로그가 frozen 을 만들어낼 경로가 생긴다 — 그건 불변식 위반이다.

**불변식 2 — `entrypoint` 는 판정에 영향 없음.** verdict 는 오직 `level` 에서 나온다.
`entrypoint: true` 는 "외부 직접 도달 표면" **메타데이터**로 카드에 노출하고, 후속 L3(콜그래프·
간접영향)의 진입점 시드로 남긴다. 이걸로 등급을 올리지 마라(등급 드리프트 = 정책 외부화 무력화).

**불변식 3 — 정책 부재/불량 시 fail-closed.** 네 계획대로다. 구체 규약:
- `policies/framework-annotations.yaml` **명시됐는데 부재** → 조용한 통과 금지, `approval_required`
  + 사유 문자열. **차단(exit 1) 금지**(2·3층 상한).
- 형제 정책 규약을 따라 **`kit/run.sh` 에서 `$POL/framework-annotations.yaml` 절대경로로 명시 배선**하고
  **preflight 필수정책 루프에도 추가**하라. — 이건 권고가 아니라 **AC 가드다**:
  TASK-029 **R-4** 가 정확히 이 누락(`--language-routing` 만 상대경로 기본값 + `run.sh` 미배선)으로
  킷 사용자 전원 상시차단을 만들었다. **같은 패턴 3회차는 없어야 한다.**
- 잘못된 YAML·중복 name·`frozen` 등급 → 검증오류로 **드러내고** 카드에 남겨라.

## 5. 질문 4 — Java `status: stub → supported` 전환

**단순 플립은 승인하지 않는다. 조건부 승인 + 스키마 확장이 필요하다.**

이유: `status` 는 **언어당 스칼라 1개**인데 `language-router.py` 는 이걸
`deep_analysis: "available"` 로 그대로 번역한다(`language-router.py:117`). J2 시점의 **사실**은:

| 층 | Java 실제 상태 (J2 완료 시점) |
|---|---|
| 인벤토리 (J1/TASK-030) | ✅ 있음 |
| 함수레벨 민감도 (J2/TASK-031) | ✅ 있음 |
| 능력 카탈로그 (J3/TASK-032) | ❌ **없음** |
| 콜그래프·간접영향 (X) | ❌ 없음 |

여기서 `supported` 로 올리면 카드가 **하지도 않은 Java 능력분석을 "available" 로 보증**한다.
`Runtime.exec` 를 새로 들인 Java PR 이 **아무 신호 없이 통과**하는데 카드는 "deep analysis available"
이라고 말한다 — **조용한 통과 + 허위 카드**로, D-074 **O-5** 가 예고한 바로 그 실패다.

**확정 요구 (TASK-031 AC 추가 — J2 활성화의 전제):**

1. `policies/language-routing.yaml` 에 **층별 `layers:`** 도입 (dev + kit 동시):
   ```yaml
   java:
     status: partial              # 코스한 요약값 (하위호환)
     layers:
       inventory:    supported
       gov_level:    supported
       capabilities: stub         # J3 에서 supported 로
       callgraph:    unsupported
   ```
   `python` 에도 동일 형태로 부여하라(스키마 균일 + Python 이 콜그래프를 과대주장하지 않게).
2. **소비자 규약**: `layers` 가 있으면 소비자는 **반드시 `layers.<layer>` 를 읽는다**.
   `supported` **정확히 일치할 때만** available 로 취급 — `partial`·미지 값은 **available 아님**(fail-safe).
   현재 `route["status"] == "supported"` 단일 비교(`language-router.py:117·120·125`)가 바뀌어야 한다.
3. **카드 coverage 는 층별로 말한다**: "java: inventory·gov_level 분석함 / capabilities·callgraph
   **미분석**". 언어 단위 이분법(checked/not_checked)으로는 이 진실을 표현할 수 없다.
4. **🔴 가용성은 정책이 아니라 실측이다 (D-074 O-5 정면 해소).** 정책이 `supported` 라고
   써 있어도 tree-sitter 가 없으면 분석은 **실제로 안 된 것**이다. `parse_error`
   (`java analysis unavailable: …`) 가 난 파일은 정책 status 와 무관하게 **카드 coverage 의
   미분석 목록에 반드시 나타나야 한다.** 이게 없으면 `supported` 전환은
   **"열화됐는데 정상이라고 말하는 카드"** 를 만든다 — J1 때는 java=`stub` 이라 무해했지만
   `supported` 로 올리는 순간 치명적이 된다. **이 항목이 status 플립의 하드 전제다.**
5. **회귀 픽스처 필수**(이 경로 현재 픽스처 0건 = 무발견이 아니라 **미측정**):
   ① tree-sitter 부재 강제 + 정책 `supported` → 카드에 **미분석 노출**·`exit 0`
   ② 층별 coverage 문구가 capabilities 를 **미분석으로 명시**
   ③ 각각 **음성검증**(노출 제거 → 해당 케이스 **단독 FAIL**).

**킷 반영**: `sync-from-dev.sh:32` 가 `policies/*.yaml` 을 통째로 복사하므로
`framework-annotations.yaml` 은 sync 로 자동 반입된다. 단 **manifest 종수·README 갱신 + dev↔kit
md5 동일성(D-068) 재확인**은 TASK-031 브랜치에서 네가 수행하라. 나는 kit 매니페스트를 건드리지 않았다
(킷 도구는 Codex 소유).
