## 🔴 R-1 (보정 사유): head 에 존재하는 분석-불능 `.py` 가 동반 레코드 하나로 **pass 세탁**

**fresh 실증(F)**: 신규 비-UTF8 `.py`(`# -*- coding: latin-1 -*-` — **합법·실행 가능한 파이썬**이며 내부에 `@gov(frozen)` 은닉 가능, 게이트는 UTF-8 디코드 실패로 내용 완전 비가시) + 같은 PR 에 watched 함수 변경 1건 →

```
verdict: pass · exit 0 · errors: [mystery.py base unreadable, mystery.py head unreadable]
```

같은 파일을 **단독 제출하면 approval_required** (`<unknown>` fail-closed 레코드) — 즉 **watched 한 건만 끼워 넣으면 분석-불능 파일이 판정에서 사라진다**(순서·동반 의존 = 세탁 경로).

- **근원**: ① 전역 fail-closed 가 `if errors and not records:` — records 가 1건이라도 있으면 발동 안 함. ② per-path fail-closed(head parse-fail 블록)는 `sensitive_levels_in_result(base)` 가 참일 때만 레코드 생성 — head 를 애초에 못 읽는 **신규/재작성 파일은 base 민감이 없어 레코드 0**.
- **위반**: AC 가드(D-017 #2) fail-closed 의 의도 = TASK-012 "빈/불능 분석 ≠ clean" 계보. §2B 필수 질문("거버넌스 목적에 직접 구멍?") → **그렇다** — head 에서 실행될 파일의 함수 거버넌스가 통째로 비가시인데 verdict 가 pass. 비차단 불가.
- **수정 계약**:
  - head 에 **존재하는** 변경 `.py` 가 `parse_error`/`unreadable` → **per-path fail-closed 레코드를 무조건 추가** (base 에 민감 주석 있으면 그 최강 레벨 — 현행 유지, 없으면 `protected`). 다른 레코드 유무와 무관.
  - head 에 파일이 **부재**(D/리네임 소스)한 경우는 구분: base 가독·민감 → 현행대로 base 최강 레벨 fail-closed(deleted-frozen 픽스처 경로) / base 가독·비민감 → **통과 허용**(양측 분석 완결 — 과차단 방지) / base 도 불가독 → `protected` fail-closed.
  - "존재 vs 부재" 판별은 결정적으로(예: `git cat-file -e <ref>:<path>` 또는 name-status A/D) — `git show` 에러 메시지 문자열 파싱 금지.
  - **top-level upstream error**(map/classify `error` 필드) → records 유무 무관 **최소 approval_required** (현행은 records 있으면 무시 — 후보 목록 자체가 불완전한데 부분 레코드로 판정 확정 금지).
