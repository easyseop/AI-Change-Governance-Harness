## R-1 (🔴 · 보정사유) — 하류 통합 회귀: 동적 watched 가 **신규 protected 정적호출을 은닉**

### 결함
TASK-016 이 `protected` 카탈로그 능력(`subprocess_exec` 등)을 **`watched` 로도** 표면화하면서, "능력 id 1개 ⇒ level 1개" 라는 하류 전제가 깨진다. `check-new-capabilities` 의 신규탐지는 **id 집합 차분**(`set(head_caps) - set(base_caps)`)만 본다 — level 은 보지 않는다. 그래서 **base 에 동적접근이 이미 있어 그 능력 id 가 `watched` 로 존재**하면, head 에서 **진짜 신규 protected 정적호출**(같은 능력 계열)이 추가돼도 id 가 이미 있어 신규로 안 잡힌다 → verdict `pass`, exit 0. **경고조차 없음.**

### 실증 (fresh, 픽스처 밖 · 실제 `policies/sensitive-capabilities.yaml`)
```
base/app/run.py:  import os
                  def f(name): return getattr(os, name)      # 동적 → subprocess_exec watched
head/app/run.py:  (위 + )  def g(cmd): os.system(cmd)          # 신규 실제 RCE → protected
```
| | verdict | exit | 결과 |
|---|---|---|---|
| **main (TASK-016 이전)** | `approval_required` | **2** | 신규 `subprocess_exec` 포착 ✅ |
| **본 브랜치 (TASK-016)** | `pass` | **0** | **은닉 — 신규 RCE 무경고 통과** ❌ |

**이는 개선이 아니라 기존 게이트의 순수 퇴행**이다. `getattr(os,name)` 류 동적 디스패처는 플러그인 로더·CLI 라우터 등에서 흔하며(base 상주 현실적), 그 뒤 실제 `os.system` 추가가 조용히 미탐된다. CLAUDE.md §2B "[비차단 판정 전 필수 질문]" — 거버넌스 목적(민감 변경 포착)에 **직접 구멍**이므로 비차단 금지.

### 보정 계약 (Codex — `check-new-capabilities.py` 만)
1. **level 에스컬레이션 탐지 추가**: 공유 능력 id(base∩head)에 대해 head level 이 base 보다 강하면(`watched`→`protected`) `new_capabilities`(또는 `escalated_capabilities` 버킷)로 넣어 **`approval_required`(exit 2)**. `LEVEL_STRENGTH` 를 추출기와 공유하거나 동일 순서를 재사용해 정합. 결정성 유지.
   - 추출기(`extract-python-capabilities.py`)·정책파일은 **건드리지 마라**. 결함은 소비자 분류 로직 1곳.
2. **상설 회귀 픽스처(고정)**: 위 실증 세트 그대로 — base `getattr(os,name)`(watched) + head `os.system`(protected) 추가 → **기대 `approval_required`·exit 2**. 반대로 base=clean → head 동적 watched 단독은 기존대로 `pass`+warning(회귀 없음 재확인).
3. **음성검증**: 위 에스컬레이션 가드를 제거하면 해당 픽스처가 `pass` 로 뒤집힘을 확인(가드 load-bearing 실증). `tests/run-tests.sh` 에 단언 추가.

> 주의: 반대 방향(base protected → head 에서 정적호출 사라지고 동적 watched 만 남음)은 **완화가 아니라 신규 능력 없음**이니 신규탐지 대상 아님(그대로 pass). 에스컬레이션(강해짐)만 잡으면 된다.
