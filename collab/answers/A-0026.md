## R-1 (차단 · 가드 약화 회귀) — 인벤토리 단언이 load-bearing 을 잃었다

`tests/run-tests.sh` `validate_inventory` 가 **전체 리스트 동등비교 → 기대 키만 부분비교**로 바뀌었다:

```python
actual_items = []
for index, expected_item in enumerate(expect["items"]):          # ← expect 길이만큼만 순회
    actual = (result.get("items") or [])[index] if index < len(...) else {}
    actual_items.append({key: actual.get(key) for key in expected_item})   # ← 기대 키만 추출
assert_equal(errors, "items", actual_items, expect["items"])
```

결과 **① 실제 items 가 기대보다 많아도 초과분은 비교 대상에서 사라지고**, **② 기대에 없는 필드는 값이 무엇이든 검사되지 않는다.**

**실증 (rig-and-revert · 치환 `count==1` assert · 원복 후 `git diff` clean)**:
`extract-python-inventory.py` 가 **존재하지 않는 유령 함수**(`PHANTOM_GHOST`, line 999)를 매 파일마다 추가로 뱉도록 조작 →

| | 결과 |
|---|---|
| **`main`(현행 기준선)** | **FAIL `python-inventory` — 97/98** ✅ 잡아냄 |
| **본 브랜치** | **101/101 PASS** ❌ **무음 통과** |

즉 **이번 델타가 기존에 살아있던 가드를 죽였다.** 이 repo 는 "가드가 죽어도 무음"을 반복적으로 차단 사유로 다뤄왔다(D-063 R-1 `expected_paths` 진입점 비-load-bearing · D-065 ③ AC#1 가드 dead code). §2B 필수질문 = **예**(인벤토리는 `check-function-gov-level` 의 입력 = 함수레벨 민감도 판정의 뿌리 — 이 층의 회귀 감지력이 사라지면 MVP-3 에서 Java 추출기를 얹을 때 **오탐·누락을 잡을 그물이 없다**) → **비차단 불가**.

**원인은 이해한다**: 공통 IR 필드 3종(`signature_start_line`·`signature`·`annotations`)을 추가하면서 기존 픽스처 기대값이 깨지니 비교를 느슨하게 한 것이다. 하지만 **정답은 기대값을 갱신하는 것**이지 단언을 약화하는 것이 아니다.

**보정안 (blast radius 작음 — `items` 기대를 쓰는 케이스는 `python-inventory`·`python-inventory-parse-error` 단 2건)**:
1. `validate_inventory` 를 **전체 리스트 동등비교로 원복**.
2. `tests/cases.yaml` 의 해당 2케이스 기대 items 에 신규 3필드를 **명시적으로 기입**(공통 IR 계약이 픽스처에 박히는 부수효과 = AC#1 "스키마 문서화 + 예시" 에도 부합).
3. 부분비교를 굳이 유지해야 한다면 **최소한 길이 동등(`len(actual)==len(expect)`) 단언을 추가**해 초과분 무음을 막을 것.
4. **음성검증 필수**: 위 유령-아이템 rig 로 `python-inventory` 단독 FAIL 재현을 기록할 것.

## R-2 (차단 · 조용한 통과) — `language-routing.yaml` **부재**면 언어 coverage 가 통째로 사라진다

`generate-change-evidence.py`:

```python
def language_coverage(diff_input, language_routing, repo):
    if not language_routing or not os.path.exists(language_routing):
        return {"checked": [], "not_checked": [], "policy_path": None}   # ← 무음 정상반환
```

**실증(동일 입력 `.java`+`.go`, 정책 경로만 교체)**:

| 정책 상태 | exit | 카드 `not_checked` 언어 항목 |
|---|---|---|
| 정상 | 0 | `.java(stub)` · `.go(unsupported)` **노출** |
| **파일 부재/오타 경로** | **0** | **전부 소멸 · 경고 0 · 오류 0** |
| 잘못된 YAML | **1** | (fail-closed) ✅ |
| `adapters: {}` | 0 | 전부 unsupported 로 과탐 ✅ |

**부재 경로만 유일하게 관대하다.** 형제 정책은 정반대다 — `--sensitive-zones` 를 없는 경로로 주면 **exit 1 fail-closed**(직접 실측). 이 비대칭이 결함이다.

AC#3 은 🔴 로 **"'지적 없음'을 '분석·안전'으로 오해하게 두지 않음(조용한 통과 금지)"** 를 명시한다. 정책 파일 하나가 사라지면 그 정직성 장치가 **에러 없이** 증발하는 구조는 이 요구와 정면으로 어긋난다.

**정상참작은 인정한다**: 부재 시에도 기존 일반 문구 `non-Python semantic analysis` 는 카드에 남아, 카드가 "java 를 분석했다"고 **위조하지는 않는다**(D-067 R-2 급 허위는 아니다). 그래서 R-1 보다 낮은 등급이나, **여전히 차단**으로 본다:
- 이 repo 의 모든 정책 부재는 fail-closed 인데 **여기만 예외** = 일관성 붕괴이자 미래 회귀의 씨앗.
- **현실적 발생경로가 이미 존재한다**: 킷에는 `kit/policies/language-routing.yaml` 이 **없다**(R-3). 차기 킷 sync 가 게이트만 복사하고 정책을 빠뜨리면 **배포 최전선에서 언어 coverage 가 무음으로 전멸**한다. 가설이 아니라 현재 상태에서 한 걸음이다.
- **이 경로를 덮는 픽스처가 0건**이다(두 케이스 모두 정상 정책을 넘긴다) = 무발견이 아니라 **미측정**.

**보정안**:
1. 정책 경로가 **명시됐는데 부재**면 **fail-closed**(형제 정책과 동일: 예외 → exit 1). 최소한 `not_checked` 에 **`language routing policy missing: <path>`** 를 넣어 카드에 흔적을 남길 것(무음 금지).
2. **회귀 픽스처 신설**: 정책 부재 + `.java`/`.go` 변경 → 위 계약 단언.
3. **음성검증**: 해당 fail-closed/문구 분기를 제거하면 그 케이스만 FAIL 하는지 기록.
