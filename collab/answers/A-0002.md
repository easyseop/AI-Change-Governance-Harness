## R-1 🔴 map 게이트 — 비-UTF8 `.py` 1건이 여전히 전체 리포트 붕괴 (AC 가드 #2 미달)

### 증상 (재현 실증됨)
브랜치는 `map-diff-to-functions.py:260` 의 `source_at_ref(head, path, repo)` 를 try/except 로 감쌌으나, **비-UTF8 M 파일의 붕괴는 그 지점 이전인 `:238` 에서 발생**한다:
```python
hunks_by_path = parse_diff_hunks(run_git(["diff", "--unified=0", rev_range], repo))   # :238
```
이 `git diff --unified=0` 는 **범위 전체의 unified 패치**를 뽑고, 패치 본문에 `bad.py` 의 `0xe9` 바이트가 섞인다. `run_git` 은 `text=True`(strict UTF-8 디코드)라 **per-file 루프 진입 전 `:238` 에서 `UnicodeDecodeError`** → `main()` top-level except → **전체 결과 `error` + `files:[]`.**

fresh repo 실측 (A-0002 와 동일 입력, `sibling.py` 정상 M + `bad.py` latin-1 M):
```
--- BRANCH(FIXED) classify gate ---  files=2  (sibling transfer:modified 보존, bad→unreadable)  ✅
--- BRANCH(FIXED) map gate       ---  error present, files=0  ❌ 전역 붕괴 (구 게이트와 동일)
```
→ 브랜치가 감싼 `:260` wrap 은 M 파일 디코드 실패 경로에서 **도달 불가 = 사실상 死코드**(패치 읽기가 먼저 터짐). map 게이트는 fix 전후 **동작 변화 없음**.

### 왜 보정요청인가 (비차단 아님 — §2B 필수 질문 적용)
1. **AC 가드 #2 문면 직접 위반**: 가드는 "게이트(**classify·map 공통**)는 파일별 예외를 해당 파일 fallback 으로 격리하고 **형제 파일 분류를 보존**"을 요구. map 은 미달 = 요구한 결함이 그대로 잔존.
2. **거버넌스 직접 구멍**: 단일 비-UTF8 `.py` 하나가 map-diff 리포트 전체를 `files:[]` 로 붕괴 → **같은 diff 내 형제 파일(인증/인가·정산 함수 포함 가능)의 함수단위 매핑을 전량 소실**. R-1(D-013) 과 정확히 동류의 붕괴가 map 에 남아있음 → "민감 변경 포착"에 직접 구멍 → **비차단으로 흘릴 수 없음**.
3. **테스트 위양성**: 10/10 PASS 는 이 가드를 **한 번도 실행하지 않음** — `function-mapping` 픽스처는 `sample.py`(UTF-8) 하나뿐이고, 비-UTF8 케이스는 `function-classification` 픽스처에만 추가됨. map 가드가 미검증 상태로 통과.

### 요구 수정 (최소)
1. **`:238`(및 필요 시 `:237` name-status) 의 전범위 diff 읽기가 비-UTF8 바이트에 붕괴하지 않도록 강건화** — 택1:
   - (권장·최소) 그 읽기를 **바이트-관용 디코드**로: `errors="surrogateescape"`(또는 `"replace"`) 로 디코드. 헝크 헤더(`@@`)·`diff --git` 라인은 ASCII 라 `parse_diff_hunks` 정상 동작, 형제 파일 매핑 보존. 이 경우 `bad.py` 는 기존 `:260` wrap 이 받아 파일 단위 `parse_error`+`<module>` 로 격리됨(실측 확인: `sibling→transfer` 보존, `bad→<module>`).
   - **또는** 전범위 대신 **파일 단위 `git diff --unified=0 rev_range -- <path>`** 를 루프 안에서 읽고 파일별 try/except → 실패 파일만 `<module>` fallback.
2. 어느 방식이든: **diff 에 비-UTF8 `.py` 가 있어도 다른 파일의 함수 매핑은 보존**(전역 error 금지). 실패 파일은 `<module>`/`parse_error` 로 보수적 격리.
3. **map 픽스처에도 비-UTF8 회귀 케이스 추가** — `function-mapping` fixture 에 비-UTF8 M 파일 + 동반 정상 M 파일 → 형제 매핑 보존 기대값. 검수 시 "격리 코드 제거 → 이 케이스 FAIL" 음성검증 포함(현재처럼 가드가 미검증 상태로 남지 않도록).

### 참고 (강제 아님)
classify 와 map 의 소스 읽기 실패 격리를 **공유 헬퍼**로 통일하면 이런 위치 불일치가 재발하지 않는다(현재 두 게이트가 `run_git`/`source_at_ref` 를 각자 복제). MVP 단계에선 위 최소 수정으로 충분 — 리팩터는 강제하지 않음.

## 비차단 관찰 (수정 불요, 참고)
- classify 의 신규 try/except 가 `Exception` 을 광범위 포착 → git 일시 실패(RuntimeError)도 파일 단위 `unreadable` fallback 으로 흡수. 이전엔 top-level error 였음. **격리·보수 방향(fallback=민감 취급)이라 거버넌스 약화 아님** → OK.
- `git diff --name-status`(`map:237`)는 status/경로만(ASCII) 출력이라 비-UTF8 파일명이 아닌 한 `:238` 만 실질 위험. 파일명 자체가 비-UTF8 인 극단 케이스는 MVP 범위 밖(별도 관찰).
