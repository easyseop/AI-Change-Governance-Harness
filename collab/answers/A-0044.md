## 2. R-2 해소 검증

### 2.1 보정 내용 (한 줄씩)

```python
def declaration_supertypes(source_bytes, node):
    names = []
    for field in ("superclass", "interfaces"):
        names.extend(type_names(source_bytes, node.child_by_field_name(field)))
    for child in node.children:                                            # ← 신규
        if child.type in {"extends_interfaces", "super_interfaces", "superclass"}:
            names.extend(type_names(source_bytes, child))
    return sorted(set(names))
```

- `node.children`(unnamed 포함) 순회 → `interface B extends A` 의 상위타입이 담긴
  **`extends_interfaces` 자식 노드**를 포착. A-0043 §3.1 이 지목한 정확한 지점이다.
- `sorted(set(...))` 이 필드조회와의 중복을 흡수 → **중복 supertype 무해**.
- 반환 계약(정렬된 simple name 리스트) 불변 ⇒ `type_relationships()` 하류 무영향.
- 그 외 로직(**`method_targets`·`collect_calls`·IR 형상) 무변경**. 엣지는 **추가 방향으로만** 움직인다.

### 🟡 차기 AC 가드 → **TASK-037 AC#6 신설** (자기정정 포함)

**O-4 익명 내부클래스 — A-0042/A-0043 의 "과대근사 방향=안전" 은 오판이었다. 실제는 누락 방향이다.**

```java
interface Port { void run(); }
class Ledger { void settle(){ } }
class Caller {
    Port p;
    void wire(){ p = new Port(){ public void run(){ new Ledger().settle(); } }; }
    void go(){ p.run(); }
}
```
실측:
```
nodes : Caller.go · Caller.run(← 익명 본문이 오귀속된 유령) · Caller.wire · Ledger.settle · Port.run
edges : Caller.go→Port.run · Caller.run→Ledger.settle · Caller.wire→Ledger.settle
sink=Caller.go forward 1/2/3/4홉 → Ledger.settle 도달? 전부 False
```
`Caller.go → Port.run` 은 **본문 없는 추상 선언에서 막다른 길**이고, 익명 구현 본문은
`Caller.run` 이라는 **연결되지 않은 유령 노드**로 떨어진다. 즉 실제 런타임 엣지를 **놓친다**.
(`Caller.wire unresolved new Port` 로 `coverage.unevaluated` 에는 남지만, A-0042 §2.3 에서 확립한 대로
그 필드는 **verdict 무기여**다.)
