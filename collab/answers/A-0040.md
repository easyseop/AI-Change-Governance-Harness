### R-1. 2층 운영자 화면 정보 손실 (`kit/run.sh:233,237`)

`check-new-capabilities` 에 `--json` 을 주면 게이트는 사람용 렌더러(`print_text`)를 **완전히 대체**한다
(`main()`: `if args.json: print(json.dumps(...)) else: print_text(result)`).
`run.sh` 는 그 JSON 을 `grep -E '"verdict":|"id":|"fail_closed":' | head -8` 로 보여준다.

**실증 ① 경로·등급 소실** (fresh repo, `requests.get` 신규 도입):
```
main   :  APPROVAL_REQUIRED: 신규 민감 능력 또는 분석 오류가 감지되었습니다.
          protected: app/net.py::outbound_network        ← 경로 + 등급 + id
branch :      "fail_closed": [],
              "id": "outbound_network",                  ← 경로 없음, 등급 없음
              "verdict": "approval_required",
```

**실증 ② protected/watched 구별 불가**: 같은 능력을 `level: watched` 로 바꿔 실행하면
`main` 은 `watched: app/net.py::outbound_network` 로 구분되지만, 브랜치는 **`"id": "outbound_network"` 로 동일**하다.
grep 이 `"level":` 를 포함하지 않아 protected·watched·shadow 가 **한 화면에서 구별되지 않는다**.

**실증 ③ `head -8` 이 verdict 를 잘라먹음**: `sort_keys=True` 라 `"verdict"` 는 JSON **뒤쪽**에 온다.
서로 다른 파일 8개가 능력을 새로 도입한 PR 에서 실제 출력:
```
    "fail_closed": [],
        "id": "outbound_network",   ← 동일 7줄, 파일 구분 불가
        …
(  "verdict": 줄은 잘려서 아예 안 보임  )
```
⇒ 운영자는 **어느 파일인지도, 승인요구인지도** 2층 블록만 봐서는 알 수 없다.

**영향 한정(정직 고지)**: `cap_exit`·최종 verdict·exit 코드는 **무손상**(위 8개 케이스에서 `rc=2` 확인).
기계 판정은 정확하다. 손상된 것은 **사람이 읽는 증거면**이며, 2층은 카드 밖이라 **대체 창이 없다**.

**보정 지시**: JSON 은 trace 용으로 유지하되 **사람용 요약을 복원**할 것.
`append_capability_trace` 가 이미 JSON 을 파싱하므로, 같은 자리에서
`level: path::id` 형태(= `print_text` 등가)를 렌더해 출력하면 게이트 2회 실행 없이 해결된다.
`head` 절단은 건수 고지(`… 외 N건`)로 대체할 것. **회귀 픽스처**: 진입점에
"protected 신규 능력 도입 → 콘솔에 **경로와 등급이 함께** 보임" 케이스 + 음성검증.

### R-2. 카드가 YAML 이 아닐 때 `append_capability_trace` 크래시 → **AC#5 의 trace 가 조용히 소실** (`kit/run.sh:172-173`)

**보정 지시**: `safe_load` 를 `try/except Exception` 으로 감싸고 `isinstance(card, dict)` 아니면
