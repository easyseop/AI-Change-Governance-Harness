## R-1: `expected-path-missing-approval` 케이스 판별력 부재 (보정요청)

### 결함

케이스가 `expected_paths` 선언(change-intent.yaml 수정)과 코드 변경을 **같은 커밋**에 실어, `HEAD~1..HEAD` diff 에 `change-intent.yaml` 자체가 포함된다. `change-intent.yaml` 은 allowed(`app/**`·`custom/**`·`policies/**`) 밖이라 **`out_of_scope:change-intent.yaml` 이 항상 동반** — exit 2 가 이중 원인이 된다(카드 reasons 실측: `out_of_scope:change-intent.yaml` 와 `missing_expected:app/required_patch.py` 병존).

카드 grep 2종도 판별력이 없다: `grep 'missing_expected:'` 는 **빈 리스트 키**(`missing_expected: []`)에 매칭되고, `grep 'app/required_patch.py'` 는 **`expected_paths` 에코**에 매칭된다 — 탐지가 안 돌아도 둘 다 충족.

### 실증 (rig-and-revert)

킷 카드 게이트의 missing 계산만 무력화(`missing_expected = []` 고정·키 에코 유지)한 rig:

| 대상 | 결과 |
|---|---|
| 진입점 스위트 | **12/12 PASS 유지** — 부재 탐지가 완전히 죽어도 침묵 |
| 러너 스위트 | `evidence-expected-missing` FAIL(82/83) → selftest 전체 FAIL |

즉 오늘은 러너 케이스가 방어하지만, **진입점 케이스 단독으로는 이 기능의 어떤 게이트측 회귀도 못 잡는다.** 제출 기록의 기대값 변조 rig(2→0 시 11/12)는 "rc 가 진짜 2"만 증명할 뿐 "2의 원인이 부재 탐지"를 증명하지 못한다.

### 왜 보정요청인가 (§2B 필수질문 = 예)

`sync-from-dev.sh` 는 `kit/tests/`(러너·cases·픽스처)를 **dev 로 덮어쓰고**, `run-entrypoint-tests.sh` 만 스냅샷 보존한다. 즉 **진입점 스위트는 sync 를 살아남는 유일한 킷 소유 가드**다. dev 쪽에서 기대값과 함께 회귀가 유입되는 시나리오(시맨틱 변경이 dev 픽스처 기대값까지 같이 바꿔 들어오는 경우)에서 마지막 독립 방어선인데, 이 기능에 대해 그 방어선이 비어 있다 → 거버넌스 목적(부재 탐지가 킷에서 산다)에 직접 구멍. AC#2 가 이 케이스를 만든 이유 자체가 무효화된다.

### 보정안 (최소 델타 — 동작 실증 완료)

1. **선언을 diff 밖으로**: `expected_paths` 선언을 base 커밋(또는 별도 선행 커밋)에 넣고, 마지막 커밋은 `app/service.py` 만 변경 → `HEAD~1..HEAD` = 코드 파일 단독. *(실증: 이 구성에서 exit 2 의 원인이 missing 단독 — out_of_scope reasons 0건. detection-kill rig 에선 exit 0 → 케이스 FAIL → load-bearing 회복.)*
2. **카드 증거 grep 강화**: `grep -q 'missing_expected:app/required_patch.py'` (reasons 라인) 하나로 — 스키마 에코와 구분되는 실제 탐지 증거를 단언.
3. 기존 기대값 변조 rig 유지 + **detection-kill rig**(카드 게이트 missing 계산 무력화 → 케이스 단독 FAIL) 실증을 handoff 에 기록.

재리뷰는 **보정 커밋만** 본다(멱등). 나머지 델타(게이트 sync·정책/템플릿·픽스처·manifest/README·selftest git init)는 이미 통과 — 재작업 불요.
