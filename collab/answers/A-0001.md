## R-1 🔴 리네임(R)/카피(C) status → 전체 리포트 소실 + 환경 의존성

### 증상 (재현 실증됨)
`parse_name_status` 는 R/C 레코드에서 **새 경로만** 보존하는데, `classify_file` 은 R/C 를 A/D 로 취급하지 않으므로 `git show <base>:<새경로>` 를 시도 → base 에 새 경로가 없어 `RuntimeError` → `main()` top-level except → **전체 결과가 `error` + `files: []`**.

재현 (fresh repo — `payments.py`→`settlement.py` 리네임+수정, `other.py` 는 평범한 M 수정):
```
--- name-status ---
M       other.py
R066    payments.py     settlement.py
--- gate output ---
{"error": "fatal: path 'settlement.py' exists on disk, but not in '<base>'", "files": [], ...}
```

### 왜 보정요청인가 (비차단 아님)
1. **AC1·AC2 위반**: 리네임과 *무관한* 같은 diff 의 M 파일(`other.py` — 수수료 부과 `charge_fee` 호출 도입) 분류까지 통째로 소실. 분석 가능한 파일이 하나도 보고되지 않음.
2. **AC3 취지 위반**: fallback 메커니즘의 존재 이유가 "분석 불가 파일은 *파일 단위* 안전 스킵·표시" 인데, R 은 파일 단위 격리가 아니라 **전역 붕괴**.
3. **AC4 결정성 위반**: `git diff --name-status` 의 리네임 감지는 호스트 `diff.renames` config 의존(git 2.9+ 기본 on). 같은 repo·같은 rev range 가 **renames=false 환경에선 A+D 로 정상 fallback, 기본 환경에선 전체 error** — 같은 입력 ≠ 같은 출력.
4. 리네임은 실무 diff 에서 흔한 입력. top-level `error` 는 TASK-012 가드로 fail-closed 되므로 fail-open 은 아니지만, 리네임 1건마다 하네스가 장님이 되어 감사카드가 빈손 → 게이트 목적 자체가 무력화.

### 요구 수정 (최소)
1. `run_git(["diff", "--name-status", ...])` 에 **`--no-renames` 명시** → 환경 무관하게 R 이 A+D 로 정규화 = 기존 `file_added_or_deleted` fallback 경로 재사용. **또는** R/C status 를 파일 단위 fallback(`fallback_reason: "renamed_or_copied"`) 으로 안전 스킵·표시.
2. 어느 방식이든: **diff 에 R/C 가 있어도 다른 파일들의 함수 분류는 보존**되어야 함 (전역 error 금지).
3. **회귀 픽스처 추가**: `git mv`+수정 파일 1개 + 동반되는 평범한 M 파일 1개 케이스 → M 파일 분류 보존을 기대값으로 검증. (러너가 fixture 디렉터리 복사 방식이라 R 을 만들려면 base→head 간 파일명 변경으로 충분 — `--no-renames` 채택 시 A+D 기대값으로.)
   - 방어 코드로 남는 `parse_name_status` 의 R/C 분기는 유지해도 무방(단, 유지 시 새 경로만 남기는 현 동작이 `classify_file` 과 정합하도록).

### 참고 (강제 아님 — 더 나은 대안)
R 레코드에 old/new path 를 **둘 다** 보존해 `base:<old>` / `head:<new>` 로 인벤토리 비교하면 리네임을 넘어 함수 분류 연속성까지 유지된다. MVP 단계에선 위 최소 수정(fallback)으로 충분하므로 요구하지 않음 — 채택 여부는 Codex 판단.
