## 2. 🔴 계약 — 소비자가 반드시 지킬 것 (AC#9(b) 구체화)

**(a) `supported` 정확일치만 available.**
`layers.<layer> == "supported"` 만 available. `partial`·`stub`·`unsupported`·**미지값**·**층 키 부재**·
**`layers:` 블록 자체 부재** = 전부 **not available**(fail-safe). 대소문자 정규화도 하지 마라 —
`Supported` 오타는 **과소주장(안전)** 으로 떨어져야지 관대하게 승격되면 안 된다.
정책에 `defaults.layer_states` 열거를 넣었으니 **열거 밖 값은 not-available + `errors` 흔적**으로 처리하라.
**조용한 통과 금지**(AC#3 계보): 어떤 층이 not-available 이면 그 사실이 카드에 **반드시 남아야** 한다.

**(b) `layers:` 부재 = 전 층 not-available.**
미동기화 킷 사본·조직 커스텀 정책이 구 스키마일 수 있다. 그때 legacy `status` 를 보고 전 층을
supported 로 역추론하면 **D-074 O-5 의 허위 카드가 그대로 부활**한다. 부재는 fail-safe 로만 접어라.

**(c) legacy `status` 는 deprecated — 소비자는 무시.**
`status` 는 **보수적 floor**(= 네 층 전부 supported 일 때만 `supported`)로 정의했다. 그래서 java 는
J2 이후에도 `status: stub` 으로 남는다. **이건 실수가 아니라 설계다**:
- 아직 `layers` 를 모르는 **구 소비자**(dev 현행 `language-router.py:117`, 미동기화 킷)가 이 정책을 읽어도
  java 를 `not_yet_available` 로 **과소주장** → 허위 카드 경로가 원천 차단된다.
- 즉 **정책만 먼저 머지돼도 판정·카드가 변하지 않는다**(§3 실측).
- ⇒ **너는 java `status` 를 `supported` 로 올릴 필요가 없다. 올리지 마라.** 층 승격은 `layers` 로만 한다.
  (`status` 를 올리는 순간 구 소비자가 있는 어떤 경로에서든 "안 한 분석을 했다"고 말하게 된다.)

**(d) 🔴 가용성은 정책이 아니라 실측 (AC#9(d) — 플립의 하드 전제).**
`layers.inventory: supported` 라도 그 파일이 `parse_error`(tree-sitter 부재·문법 부재·문법오류)면
**카드 미분석 목록에 반드시 노출**하라. 정책은 *의도*, 카드는 *실제*를 말해야 한다. D-074 R-1 이
정확히 이 계열이었다(의존 부재인데 정책은 정상). 정책 층 상태와 실측이 어긋나면 **실측이 이긴다.**

**(e) 카드 coverage 는 층별 서술 (AC#9(c)).**
`deep semantic analysis not yet implemented for .java` 같은 **언어 단위 이분법 문구는 이제 거짓**이다
— J2 java 는 인벤토리·등급을 실제로 했다. 최소한 이 형태로 나눠라:
```
java: inventory ✅ · gov_level ✅ · capabilities ❌(미구현) · callgraph ❌(미구현)
```
특히 **capabilities 미분석은 명시적으로 문장에 남을 것** — `Runtime.exec` 신규 도입 Java PR 이
무신호로 통과하는 현실을 승인자가 알아야 한다. 이게 AC#9 의 존재 이유다.

**(f) verdict 무영향.**
층 상태는 **coverage/정직성 층**이다. 어떤 층이 stub 이라고 verdict 를 올리지 마라(과차단),
supported 라고 내리지도 마라. 판정은 기존 경로층·`level` 에서만 나온다(D-075 불변식 2 와 동형).
