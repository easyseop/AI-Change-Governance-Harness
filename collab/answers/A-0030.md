## 🔴 R-1 (차단) — tree-sitter 의존 부재/불량 시 `.java` 포함 diff 가 **전량 🔴 BLOCKED** + 허위 사유 + **Python 매핑 전멸**

### 증상 A — 판정 회귀 (0 → 1)

동일 repo · 동일 무해 변경(`.java` 파일 1개 추가) · `import tree_sitter` 실패 shim:

```
main   kit/run.sh → 🟢 PASS  (exit 0)
branch kit/run.sh → 🔴 BLOCKED (exit 1)
```

**`tree_sitter` 자체는 정상이고 `tree_sitter_java` 문법만 없어도 동일하게 BLOCKED** —
킷 `requirements.txt` 부분설치는 현실 배포 경로다. `run.sh` 에는 `check-tree-sitter-languages`
배선이 **0건**(`grep -c` = 0)이라 **사전 진단 경로도 없다**.

### 증상 B — 가상 시나리오가 아니라 이미 터짐

내 실행환경에서 킷 진입점 스위트가 **3/3 재현적으로 FAIL**
(`language-routing-kit-policy-default`: `exit=1 expected=0`). **main 은 같은 조건 3/3 PASS.**
카드 `reasons` 에 찍힌 실제 원인:

```
dlopen(.../tree_sitter/_binding.cpython-311-darwin.so):
  (mach-o file, but is an incompatible architecture (have 'arm64', need 'x86_64'))
```

즉 **아키텍처 불일치라는 환경 결함이 곧 거버넌스 차단**이 된다.
(참고: `ACGH_GATE_TIMEOUT_SECONDS` 를 1→8 로 올려도 그대로 FAIL ⇒ 타임아웃 문제 아님.
같은 repo 를 스위트 밖에서 다시 돌리면 PASS ⇒ 게이트 로직이 아니라 **의존 해소 경로**의 문제.)

### 증상 C — 사유 정직성 붕괴 (승인자를 오도)

차단 카드가 이렇게 나온다:

```yaml
sensitive_zone_check: {status: blocked, frozen_touched: [], protected_touched: []}
verdict: blocked
coverage_statement: {verdict_statement: governance violation detected}
reasons: ['dlopen(...) incompatible architecture ...']
```

**민감영역은 하나도 안 건드렸는데 "governance violation detected"** 로 표시된다.
승인자가 보는 사유는 dlopen 오류 문자열뿐 — TASK-029 R-4 에서 확인한 "상시 🔴 = 신호 정보량 0" 과
같은 구조이며, 사유가 **허위**라는 점에서 한 단계 더 나쁘다.

### 증상 D — 반대방향 구멍: **같은 diff 의 Python 함수 매핑까지 전멸** (under-detection)

`extract_java_inventory` 의 `ImportError` 는 `map-diff-to-functions` 의 캐치 튜플
(`RuntimeError, UnicodeDecodeError, UnicodeEncodeError`) **밖**이라 `main()` 의
전역 `except Exception` 으로 빠져 **`files: []`** 를 반환한다.

```
정상 환경 : [('services/settlement/X.java', ['X']), ('services/settlement/calc.py', ['calc'])]
깨진 환경 : files: []            ← 정산 경로 Python 함수 변경이 매핑에서 사라짐
```

`.java` 파일 **하나**가 그 diff 의 **모든 Python 파일** 함수레벨 추적을 무력화한다.

### §2B 필수질문 = **예** (양방향)

과차단(무해 변경 상시 🔴 + 허위 사유)과 과소탐지(Python 함수 매핑 전멸)를 **동시에** 낸다.
거버넌스 목적에 직접 구멍이므로 **비차단으로 흘릴 수 없다**.
AC#4 는 "파싱 실패 → **파일 단위 격리**, 형제 보존, 빈 인벤토리 + 오류표시, exit 0" 을 요구하는데
**의존 부재는 문법오류보다 더 흔한 실패**인데도 이 격리 규약 밖에 있다.

### 보정안 (blast radius 작음 — 게이트 2파일 + 픽스처)

1. `load_java_inventory_module()` / `extract_java_inventory()` 의 **`ImportError`·`OSError` 를 잡아
   파일 단위 격리로 강등**: `{"items": [], "parse_error": "java analysis unavailable: <err>"}`.
   → `classify` 는 기존 `parse_error` → `fallback_file` 경로를 그대로 타고,
   `map-diff` 는 `file_record["parse_error"]` 로 정직하게 노출된다(둘 다 이미 배선돼 있음).
2. `map-diff`·`classify` 의 except 튜플에 `ImportError`·`OSError` 추가 —
   전역 `except Exception` 으로 **`files: []` 가 되는 경로를 봉쇄**(증상 D).
3. 그 결과가 **차단이 아니라 coverage 노출**로 흐르는지 확인. `.java` 는
   `language-routing.yaml` 상 `status: stub` 이므로 `not_checked` / "deep semantic analysis not yet
   implemented" 계열로 나가야 하고 **verdict 상승 금지**.
4. **회귀 픽스처 신설**(현재 이 경로 픽스처 **0건** = 무발견이 아니라 **미측정**):
   의존 부재를 강제한 상태에서 ① `.java` 포함 diff → **exit 0 · 카드에 coverage 문구**,
   ② python+java 혼합 diff → **Python 함수 매핑이 살아있음** 을 각각 단언.
5. **음성검증**: ①의 격리를 원복하면 위 케이스가 **단독 FAIL** 하는지 실증.
6. (권장) `run.sh` preflight 또는 카드에 "Java 심층분석 가용 여부"를 1회 진단해 coverage 에 명시.
   현재 사전 진단 경로가 없어 사용자는 dlopen 오류를 보고서야 알게 된다.
