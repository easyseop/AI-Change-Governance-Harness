## 3. 🔴 R-4 — **중첩 deferred body 에서 AC#6 가드가 스스로 무력화** (보정이 만든 회귀)

### 3.1 실증 (픽스처 밖 fresh 입력 · 익명클래스가 람다 본문 **안**에 있음)

```java
interface Port { void run(); }
interface Inner { void go(); }
class Ledger { void settle() { int v = 1; } }   // ← head 에서 v = 2
class Flow {
    Port port;  Inner in;
    void sink() { port.run(); }                  // ← 등록 sink (enforcing, hops 4)
    void wire() { port = () -> { in = new Inner() { public void go() { new Ledger().settle(); } }; in.go(); }; }
}
```

런타임 경로는 **실재한다**: `Flow.sink → port.run() → [람다 본문] → in.go() → [익명 본문] → Ledger.settle`.

| | 엣지 | `coverage.unevaluated` | verdict |
|---|---|---|---|
| **보정 전** | `Flow.go→Ledger.settle`, `Flow.sink→Port.run`, `Flow.wire→Inner.go`, `Flow.wire→Ledger.settle` | `anonymous_class` **있음** | **`approval_required` · exit 2** |
| **보정 후** | `Flow.sink→Port.run`, `Port.run→Inner.go` — **여기서 막다른 길** | **완전히 빔 `[]`** | **`pass` · exit 0** |

**보정 전에는 잡히던 실제 도달 경로가 보정 후 조용히 통과한다.** verdict 회귀 + 정직성(coverage) 회귀가 동시에 발생했다.

### 3.2 원인 (한 줄씩)

- `nested_in_deferred_body(node, body)` 가 `collect_calls` 에서 **중첩 본문 안의 모든 노드를 제거**한다 — 정의부 오귀속을 없애려는 의도는 옳다.
- 그런데 대체 처리기인 `subtree_call_targets()` 는 `method_invocation` 과 `object_creation_expression` **두 분기밖에 없다**. `lambda_expression` 분기도, 중첩 익명 처리도 없다.
- 게다가 `if not resolved and owner_type and not has_anonymous_class_body(node)` 때문에 중첩 익명은 **`unresolved` 기록조차 남기지 않는다**.
- 결과: **깊이 2 이상의 dispatch 는 엣지도 흔적도 없이 증발**한다. AC#6 이 세운 `anonymous_class`→fail-closed 가드는 **자기 자신이 만든 스킵 때문에 발동하지 못한다**.

동일 형태의 **람다-in-람다**(ADV-4)도 `coverage` 가 보정 전 1건 → **보정 후 `[]`** 로 퇴화한다(verdict 는 양쪽 `pass`).
MVP-3 공통규칙 "미해소 호출은 **coverage 정직 노출**(조용한 통과 금지)" 위반이다.

### 3.3 §2B 필수질문 — **예**

"거버넌스 목적(민감 변경 포착)에 직접 구멍을 내나?" → **예.** 실제 sink 도달 변경이 `exit 0` 이고, 게다가 **보정 전보다 나빠졌다**. 비차단 금지.

## 4. 🔴 R-5 — `lambda_dispatch` 가 **fail-closed 승격 경로에 미편입** (A-0045 §2 (b) 요구의 반쪽)

A-0045 §2 는 차선책을 "`kind: lambda_dispatch` 로 남기고 익명과 **같은 fail-closed 승격 경로에 편입**" 이라고 못박았다.
보정은 **`kind` 만 만들고 승격을 배선하지 않았다** — `check-indirect-impact.py` 의 승격 필터는 여전히 `kind == "anonymous_class"` 단일 비교다.

### 4.1 실증 — dispatch 대상 타입을 못 정하는 두 최빈 형태

```java
// ADV-2b  람다를 메서드 인자로 전달 (executor.submit / forEach / 빌더 콜백 = Java 최빈형)
void sink() { port.run(); }
void set(Port p) { port = p; }
void wire() { set(() -> new Ledger().settle()); }

// ADV-6  default 메서드를 가진 함수형 인터페이스 (Comparator·Function 계열의 정상 형태)
interface Port { void run(); default void helper() { int x = 0; } }
void wire() { port = () -> new Ledger().settle(); }
```

| 입력 | `coverage` | `fail_closed` | verdict |
|---|---|---|---|
| ADV-2b | `lambda_dispatch` **기록됨** | **없음** | **`pass` · exit 0** |
| ADV-6 | `lambda_dispatch` **기록됨** | **없음** | **`pass` · exit 0** |

ADV-6 의 원인은 `functional_method_targets()` 의 `if len(methods) == 1` 이 **`default`·`static` 메서드까지 세기** 때문이다.
`default` 헬퍼를 가진 함수형 인터페이스는 표준적이므로(`Comparator.reversed()` 등) 실무에서 흔히 걸린다.
**방향이 안전 쪽이 아니다** — 대상을 못 정하면 `pass` 로 떨어진다.

### 4.2 수정 비용 — **1줄 · 회귀 0** (직접 실증)

```python
# .harness/gates/check-indirect-impact.py
- if item.get("kind") == "anonymous_class"
+ if item.get("kind") in {"anonymous_class", "lambda_dispatch"}
```

이 1줄만으로 **ADV-2b·ADV-6 이 `approval_required`(2) 로 전환**되고, 정상 탐지 케이스(ADV-1)는 실 발견을 그대로 유지하며, **스위트 157/157 불변**(직접 적용·확인). 대규모 리팩터가 아니다.

(참고: R-4 는 이 1줄로는 닫히지 않는다 — `subtree_call_targets` 쪽 작업이 필요하다. Claude 가 5줄짜리 평면 마커 스케치를 시도했으나 기존 2케이스를 깨뜨렸으므로, **평면 마커보다 재귀 처리 쪽이 맞다**고 본다. 이 스케치는 해법 제시가 아니라 **비용 하한 확인**용이며, 실패한 사실을 그대로 적어 둔다.)

## 5. 구조적 관찰 — 왜 R-4·R-5 가 동시에 생겼나

보정은 dispatch 를 잇기 위해 **정의부 귀속 엣지(`Flow.wire → Ledger.settle`)를 제거**했다. 그 자체는 정확도 개선이다
(ADV-7 실증: sink 가 `wire()` 만 부르고 람다를 **호출하지 않는** repo 에서 보정 전 과탐 1건이 사라진다 — 런타임 미도달이므로 `pass` 가 옳다).

문제는 그 **보수적 fallback 을 걷어냈는데 대체 처리가 전면적이지 않다**는 점이다.
새 처리가 커버하는 건 **① 깊이 1 · ② 직접대입(`x = ...` / `T x = ...`) · ③ 메서드 정확히 1개인 인터페이스** 뿐이고,
그 밖은 과탐이 아니라 **누락**으로 떨어진다 — MVP-3 공통규칙의 "**항상 과탐 쪽으로 반올림 · 과소탐 금지**" 와 반대 방향이다.

⇒ 보정 방향(a 노선)은 옳다. 다만 **fallback 을 제거한 만큼 dispatch 대상 미확정을 전부 fail-closed 로 회수**해야 등가가 성립한다. R-5 의 1줄이 그 회수 장치이고, R-4 는 그 장치가 도달하지 못하는 사각이다.
