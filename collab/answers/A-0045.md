## 2. 🔴 R-1 — 람다 dispatch 가 익명클래스와 **동일 구조인데 조용히 `pass`** (AC#6 미충족)

### 실증 (픽스처 밖 fresh 입력, 두 파일은 `wire()` 한 줄만 다름)

```java
// ADV-1  람다
interface Port { void run(); }
class Ledger { void settle() { int v = 1; } }   // ← head 에서 v = 2 로 변경
class Flow {
    Port port;
    void sink() { port.run(); }                 // ← 등록 sink (enforcing, hops 4)
    void wire() { port = () -> new Ledger().settle(); }
}
// ADV-1b  익명클래스 (wire() 만 교체)
    void wire() { port = new Port() { public void run() { new Ledger().settle(); } }; }
```

| 입력 | verdict | exit | fail_closed |
|---|---|---|---|
| ADV-1b 익명클래스 | `approval_required` | **2** | `anonymous class dispatch` |
| ADV-1 **람다** | `pass` | **0** | **없음** |

두 경우 모두 런타임에 `Flow.sink → port.run() → Ledger.settle` 로 **실제 도달한다**.
바뀐 함수(`Ledger.settle`)는 enforcing sink 로부터 실제로 도달 가능한데 람다 쪽만 **exit 0 · `coverage` 무표식 · `fail_closed` 없음** 이다.
= AC#6 이 금지한 **"조용한 `pass`"** 그 자체, 그리고 MVP-3 공통규칙의 **"놓침 = parity 위반"** 에 해당한다.

### 픽스처가 이 구멍을 못 잡는 이유 — 람다 줄은 **장식**이다

`tests/fixtures/java-indirect-anonymous/*/app/Flow.java` 에 `lambdaWire()` 가 들어있지만
**어떤 단언도 람다를 구별하지 않는다**. 실증(**RIG-G**): 그 픽스처에서 익명 `wire()` 줄만 지우고 **람다만 남기면**
`java-indirect-anonymous-fail-closed` 가 **단독 FAIL**(150/152) 하고 게이트는 `pass` 를 낸다.
즉 이 케이스의 `approval_required` 는 **100% 익명클래스에서만** 나온다.

### 기록의 정직성 문제

`handoff-log` 와 커밋 메시지는 "람다는 `Flow.lambdaWire→Ledger.settle` 엣지 보존 확인" 이라 적었다.
그 엣지가 보존되는 건 **정의부**(람다 본문이 바깥 메서드에 귀속)일 뿐이고, AC#6 이 물은 건 **dispatch 도달성**이다.
`port.run()` 은 본문 없는 `Port.run` 에서 막다른 길이라 **정의부 엣지는 sink 경로에 연결되지 않는다**.
AC#6 은 "닫든 관찰로 남기든 **명시**" 를 허용하지만, 이 서술은 명시가 아니라 **긍정 오인을 유도**한다.

### 요청

**택1로 닫아라.**
- **(권장·AC#6(a) 노선)** 익명 본문/람다 본문을 **대상 함수형 인터페이스의 구현체로 등록**해 `Port.run → <본문>` 엣지를 잇는다. 이러면 R-1 과 O-1(아래 소음)이 **동시에** 해결된다.
- (차선·AC#6(b) 노선) 람다도 `coverage.unevaluated` 에 `kind: lambda_dispatch` 로 남기고 익명과 **같은 fail-closed 승격 경로**에 편입한다.

**상설 회귀 픽스처 + 음성검증 필수**: 위 ADV-1 형태(람다 **단독**, 익명 없음)를 별도 케이스로 고정하고,
처리 제거 시 **그 케이스가 단독 FAIL** 함을 확인할 것. 현행처럼 익명 케이스에 람다 줄만 얹는 방식은 **검증력 0** 이다(RIG-G).

### 3.3 실증 B — **부분 드리프트 = 침묵하는 반쪽 누락** (가장 위험)

`.py` + `.java` 를 함께 바꾼 커밋에 `.py` 만 라우팅에서 빠진 정책을 주면:

| `.py` 드리프트 | `approval_required` | `java_sink` — **`py_sink` 소실** |

verdict 는 그대로 `approval_required` 라 **겉보기엔 정상**이다. 소실된 건 Python 발견 1건과 **`py-reviewer` 라우팅**뿐 —

### 3.4 실증 C — **분석 대상 repo 가 자기 게이트를 끈다** (`main` 대비 회귀)

| 게이트 | verdict | exit | 탐지 |
|---|---|---|---|
| **`main` 의 게이트**(라우팅 무의존) | `approval_required` | **2** | `money` (settlement-reviewer) |

## 4. 🟠 R-2 — 다국어 repo 에서 **거짓 `unresolved_registry_function`** → `fail_closed` 상시 점등

`extract_sinks(repo, ..., languages)` 가 **함수 수집만** 언어로 게이팅하는데 **registry 검증은 전역**이라,
비활성 언어의 sink 등록이 전부 "해소 불가" 로 찍힌다.

실증 — registry 에 Python sink 1 + Java sink 1 을 둔 repo:

| 변경 | verdict | errors |
|---|---|---|
| `.java` 만 | `approval_required` | `unresolved_registry_function: app.flow.sink` ← **거짓** |
| `.py` 만 | `approval_required` | `unresolved_registry_function: Flow.sink` ← **거짓** |
| 둘 다 | `approval_required` | 없음 |

방향은 안전(과대)하지만 **틀린 사실을 감사 출력에 적는다** — registry 는 멀쩡한데 "sink 등록이 깨졌다" 고 보고한다.
그리고 다국어 repo(=MVP-3 의 표적)에서는 **모든 단일언어 커밋이 상시 `fail_closed`** 라 이 채널의 신호가 죽는다.
`fail_closed` 는 진짜 분석실패를 알리는 마지막 채널인데, 상시 점등되면 R-1·R-3 같은 진짜 사고를 덮는다.

**요청**: registry 해소용 `functions_by_name` 은 **활성언어와 무관하게** 채우거나(성능 최적화를 정확성보다 앞세우지 말 것),
registry 검증 범위를 활성언어 sink 로 한정하라. 둘 중 무엇이든 **다국어 repo 단일언어 커밋 → `errors` 비어있음** 케이스로 고정.
