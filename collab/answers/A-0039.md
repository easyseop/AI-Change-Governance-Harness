## 4. 적대적 검증 (깨보려 한 기록)

### 4.1 rig-and-revert — 보정이 load-bearing 인가
- **R1**: 킷 3줄을 옛값으로 되돌림 + 진입점 grep 도 옛값 → `kit/selftest.sh --quick` =
  **cases 135/139**(정확히 그 3건) + **진입점 21/22** ⇒ D-081 RED **정확 재현**.
  보정을 되돌리면 다시 깨진다 = 4곳 모두 **실효 가드**.
- **R2**(정책 회귀 rig): 킷 정책만 `capabilities: stub` 으로 되돌리고 나머지는 합동 상태 유지 →
  **cases 3건 FAIL(135/139)** 로 **잡힌다** ⇒ 킷 자체시험이 정책 회귀를 감지.
  단, **진입점은 22/22 로 통과** — §6 O-1 참조(비차단 관찰).

### 4.2 픽스처 밖 fresh 입력 (독립 검증 · rigged 차단)
새 git repo(`app/Payment.java`)를 만들어 `Runtime.getRuntime().exec` + `Class.forName`/
`Method.invoke` 를 head 에 도입 → `kit/run.sh HEAD~1..HEAD`:
- 2층 `protected: app/Payment.java::command_exec` · `::reflection` 감지 → **exit 2 (승인필요)**
- 카드 실물: `checked: 'java .java layers available: capabilities, gov_level, inventory'` /
  `not_checked: 'java .java layers not available: callgraph'` ⇒ **승격 후 문구가 사실과 일치**
  (capabilities 는 실제로 분석됐고, callgraph 만 미분석).

### 4.3 fail-closed 전 경로 (fresh 재현 · 킷 지침 (3))
| 경로 | 재현 방법 | 결과 |
|---|---|---|
| 분석기 부재 | `PYTHONPATH` 로 `tree_sitter` ImportError 심 주입 | 2층 `fail_closed: head file could not be parsed` → **exit 2** |
| 정책 손상 | `--policies` 오버라이드 디렉터리의 `java-sensitive-capabilities.yaml` 을 깨진 YAML 로 | 2층 `fail_closed: capability analysis failed` → **exit 2** |
| `HAS_RANGE=0` | 단일 ref(`HEAD`)로 호출 | 능력·간접영향·정책 층 전부 `range_required` → **exit 2** |
| 필수 정책 파일 부재 | `--policies` 디렉터리에서 `language-routing.yaml` 삭제 | preflight `✗ 필수 정책 파일 없음` → **exit 2**(차단 카드 아님) |
| 정책 디렉터리 부재 | `--policies <없는 경로>` | preflight `✗ 정책 디렉토리 없음` → **exit 2** |
⇒ 승격이 **fail-closed 경로를 어느 것도 약화시키지 않음**을 fresh 입력으로 확인.

### 4.4 킷 sync 완결성 — 개수 아닌 실작동 (킷 지침 (4))
합동 상태에서 `kit/sync-from-dev.sh` 를 **실제로 재실행** → 게이트 20/20, **추가 드리프트 0**
(`git status` 에 kit 변경 없음). 즉 킷은 dev 의 **충실한 스냅샷**이다.
- **역방향 실증(왜 "합동" 이어야 하는가)**: dev 정책을 stub 인 채로 두고 sync 를 돌리면
  `kit/policies/language-routing.yaml` 이 **stub 으로 되돌아간다**(실측). 즉 브랜치만 단독
  머지하면 다음 sync 때 킷 승격이 **조용히 원복**되고 킷 기대값과 다시 모순 → RED 재발.
  ⇒ (c) 정책 승격 동반 머지는 **선택이 아니라 필수**임을 실증으로 확정.

### 4.5 하류 영향 (다음 태스크가 이 출력을 어떻게 쓰나)
- `language-routing` 소비처는 `generate-change-evidence.language_coverage()` **단 하나**
  (+ `language-router` 자신). `check-new-capabilities` 는 라우팅을 **읽지 않고** 확장자로
  Java 추출기를 고른다 ⇒ 승격은 **게이트 실행 게이팅을 바꾸지 않는다**. 승격의 blast radius =
  카드 coverage 문구.
- `deep_analysis: available` 승격은 **전 층 supported** 일 때만 → java 는 callgraph stub 이라
  여전히 `not_yet_available`. **과대주장 없음**.
- `java.status` 는 `stub` 유지 = D-076 under-claim 계약(레거시 하한). AC#11 의 "status 하한
  재평가" 결과 = **변경 없음이 정답**.

## 6. 비차단 관찰 → 차기 AC 가드 (거수기 방지 · 명시 기록)

세 건 모두 **판정(차단/승인) 결과를 바꾸지 않음**을 실증으로 확인했기에 비차단으로 두되,
"후속 책임" 으로 흘리지 않고 **차기 태스크 AC 가드로 명시**한다.

- **O-1 (킷 진입점 grep 이 접두 부분일치 → 층 회귀 맹점)**
  `kit/tests/run-entrypoint-tests.sh:138` 의 `grep -q 'java .java layers not available: callgraph'`
  는 **부분일치**라 회귀 문자열 `...callgraph, capabilities` 도 매치한다.
  실증(R2): 킷 정책을 stub 으로 rig 해도 **진입점 22/22 통과**(cases 3건이 잡아 킷 전체는 RED).
  ⇒ **차기 AC**: 진입점 케이스는 `checked: ... layers available: capabilities, gov_level, inventory`
  **양성 문자열까지 단언**하거나 종단 고정 매치로 바꿔, 층 회귀를 진입점 단독으로도 잡을 것.
  *(구멍 판단: 킷 자체시험 전체로는 회귀가 잡히므로 거버넌스 포착에 직접 구멍 아님.)*

- **O-2 (D-076 trace 계약 비대칭 — capabilities 실패가 카드에 흔적 없음)**
  `capabilities: supported` 선언 후, **런타임에 Java 능력 추출이 실패한 실행**에서도 카드는
  `layers available: capabilities...` 라고 적는다. gov_level 은 실패 시
  `java gov_level analysis unavailable for <path>: ...` 를 coverage 에 남기는데(계약 준수),
  **capabilities 는 대응 흔적이 없다**(카드가 2층 결과를 싣지 않기 때문).
  실증: 심 주입 fresh 실행에서 카드에 gov_level 흔적만 존재. **단 판정은 안전** —
  `run.sh` 2층이 `fail_closed` 로 **exit 2** 를 올린다(§4.3).
  ⇒ **차기 AC(다음 Java/카드 태스크)**: 정책 헤더가 요구하는 "실패는 coverage 에 흔적을 남긴다"를
  capabilities 층에도 적용 — 카드 coverage 에 `java capabilities analysis unavailable for <path>: ...`
  추가(gov_level 대칭).

- **O-3 (킷 manifest 메타데이터가 J3 를 미반영 — 선재)**
  `kit/manifest.yaml` 의 `version: "0.3.2-mvp3-j2"` · `reflects_mvp: [... MVP-3-J2]` 가
  **J3(TASK-032) 반입 후에도 J2 로 남아 있다**(D-080 머지 시점부터 선재). 게이트 목록 자체는
  **20/20 완전**(`extract-java-capabilities` 포함)이라 실행·조립에는 영향 없음.
  원인: `sync-from-dev.sh` 가 manifest 를 갱신하지 않는데 주석은 "sync 로 갱신" 이라고 적혀 있음.
  ⇒ **차기 AC(킷 태스크)**: sync 가 manifest `version`/`reflects_mvp` 를 갱신하거나,
  selftest 가 "manifest 선언 ↔ 실제 반입 층" 불일치를 잡는 가드를 둘 것.

- **O-4 (사소 · 주석 문구 스테일)** dev/kit `language-routing.yaml` 의
  `status: stub  # deprecated floor — capabilities/callgraph not implemented` 주석이 이제
  부정확(capabilities 는 구현됨). **dev↔kit 바이트 동일 유지를 우선**해 이번엔 손대지 않았다
  (킷 파일은 Codex 소유). 차기 킷 sync 태스크에서 dev·kit 동시 갱신 권장.
