### 2.2 🔴 R-5 (차단) — `method_reference` 인자전달 갈래 **회귀보호 0**

추출기에 **쌍둥이 블록 두 개**가 추가됐다. 람다 쪽은 픽스처 2건이 고정하는데(RIG-3), **`::` 쪽은 어느 케이스도 단언하지 않는다.**

**RIG-4** — `method_reference` 갈래의 `if owner_from_invocation: unresolved.add(...)` **9줄만** 제거:

```
Summary: 198/199 PASS      ← FAIL 0건. 스위트가 전혀 눈치채지 못한다.
```

그런데 **fresh 입력은 조용히 뚫린다.** `hops: 1` 로 보수 인접이 닿지 못하게 하면 이 레코드가 **유일한 방어선**이다:

```java
interface Task { void exec(); }
class Vault { void transfer() { int amount = 1; } }          // ← 변경 대상
class Registry { void put(String k, Task t) {…} Task find(String k) { return null; } }
class Flow {
    Registry reg = new Registry();  Vault vault = new Vault();
    void wire()          { reg.put("pay", vault::transfer); }   // ← 인자전달 ::
    void mid(String k)   { reg.find(k).exec(); }                // ← 불투명
    void sink(String k)  { mid(k); }                            // sink, hops: 1
}
```

| `main` | `approval_required` (2) | `1 deferred dispatch; dead_ends=Flow.mid` | `Flow.wire\|method_reference\|vault::transfer` 포함 |
| 브랜치 | `approval_required` (2) | (있음) | 포함 |
| **RIG-4** | **`pass` (0)** | **`[]`** | **레코드 증발** |

⇒ **A-0051 R-2 와 완전히 동종**(그때는 교집합 축이 `184/184` 무변화라 차단했다). 지금 코드는 옳지만 **무방비**라 다음 편집 한 번에 조용히 되돌아간다. §2B 상설 회귀 픽스처 원칙 위반.
**보정**: 위 형태를 픽스처 1건으로 고정 + **음성검증 단독 FAIL** 확인. (람다 twin 은 `hops:4` 로도 보수 인접이 덮어버리므로 **`hops:1` 로 인접 경로를 굶겨야** 이 갈래만 단독 단언된다 — 위 입력이 그 설계다.)

### 2.3 🔴 R-6 (차단) — `inferred: true` 가 **관측된 경로**에도 찍힌다

AC#10 은 명시했다: *"보수 추정으로 추가된 홉은 **실제 관측 호출과 구분 표시**할 것."* 현행 구현은 구분하지 못한다.

```python
for callee in lambda_bodyless:
    adjacency.setdefault(caller, [])
    if callee not in adjacency[caller]:
        adjacency[caller].append(callee)
        adjacency[caller].sort()
    inferred_edges.add((caller, callee))     # ← if 블록 **밖**: 이미 관측된 엣지도 추정으로 등록
```

**fresh 실증(P6)** — sink 안에 **관측 호출**(`direct.exec()`)과 불투명 호출(`reg.find("x").exec()`)이 함께 있는 형태:

```java
class Flow {
    Registry reg = new Registry();  Task direct;
    void wire() { reg.put("pay", () -> new Vault().transfer()); }
    void sink() { direct.exec(); reg.find("x").exec(); }
}
```

추출기 `edges` 가 두 홉을 **모두 실제 호출로** 기록한다:
```json
{"call":"direct.exec","caller":"Flow.sink","callee":"Task.exec","line":8}
{"call":"lambda",     "caller":"Task.exec","callee":"Vault.transfer","line":7}
```
그런데 카드는 `path: [Flow.sink, Task.exec, Vault.transfer]`, **`inferred: true`** 로 렌더한다 — **추정 홉이 0개인 완전 관측 경로**를 보수 추정으로 표시한다.

**거버넌스 영향**: verdict 는 안전하다(`approval_required` 유지). 그러나 감사카드는 **사람이 30초 안에 위험을 판단하는 물건**(CLAUDE.md §2C)이고, 이 표식의 존재 이유는 *"이건 확실치 않으니 감안하라"* 다. 실제로 관측된 민감 경로가 "추정" 으로 내려가면 **리뷰어가 진짜 위험을 후순위로 밀 근거를 준다.** AC#10 이 금지한 방향(추정→실경로처럼)의 **거울상**이며, 표식의 신뢰성을 통째로 무의미하게 만든다.

**보정(리뷰어가 실측)** — `inferred_edges.add(...)` 를 `if` 블록 **안으로** 1줄 들여쓰면 된다:

| | P6 카드 | 스위트 |
|---|---|---|
| 현행 | `inferred: true` ❌ | 198/199 |
| **수정 후** | **`inferred` 없음** ✅ | **198/199 — `…registry-lookup-lambda` 의 진짜 `inferred: true` 도 그대로 유지** |

**동시에 회귀보호도 0이다**: 올바른 구현과 버그 구현이 **둘 다 198/199** ⇒ 스위트는 표식의 *존재* 만 고정하고 *정확성* 은 고정하지 않는다. **P6 형태를 픽스처로 추가**해 "관측 경로에는 `inferred` 가 붙지 않는다" 를 단언하고 음성검증할 것.

## 4. 🟡 O-25 (신규 · 비차단이나 함께 닫을 것) — `fail_closed` detail 의 `dead_ends=` 가 **빈 문자열**
