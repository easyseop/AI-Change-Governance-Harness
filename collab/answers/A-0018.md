## 🟠 R-1 (블로킹) — bare 모듈함수 호출이 동명 클래스메서드로 오해소 → 진짜 엣지 유실

**위치**: `.harness/gates/extract-callgraph.py` `CallVisitor.visible_local_names()` (231~237행) + `resolve_repo_function()` 후보선택(229행).

**무엇이 잘못됐나**: `visible_local_names` 가 **모듈 내 모든 클래스**에 대해 `f"{class_name}.{name}"` 후보를 **무조건** 추가한다. `resolve_repo_function` 은 후보 중 `sorted(set(candidates))[0]`(사전순 최소)를 고른다. 대문자 클래스명(`C`=0x43)이 소문자 함수명(`f`=0x66)보다 **항상 먼저** 정렬되므로, 같은 모듈에 **모듈함수 `foo` 와 클래스메서드 `C.foo` 가 공존**하면 bare `foo()` 호출이 **클래스메서드로 뒤바뀐다**.

```python
    def visible_local_names(self, name):
        names = [name]
        for index in range(len(self.parents), 0, -1):
            names.append(".".join(self.parents[:index] + [name]))
        for class_name in sorted(self.class_names):   # ← 무조건 클래스 확장 (버그원)
            names.append(f"{class_name}.{name}")
        return names
```

Python 에서 **bare 이름은 클래스멤버에 절대 바인딩되지 않는다**(`self.foo()`/`C.foo()` 필요). 즉 이 클래스 확장 루프는 **정상 엣지를 하나도 만들지 못하면서**(공식 fixture 엣지 7개 전부 이 루프 없이도 동일 산출 — 아래 실증), **진짜 엣지만 오염**시킨다.

**fresh 적대입력 실증(픽스처 밖)** — `adv1/app/m.py`:
```python
def foo():        return "module-foo"
class C:
    def foo(self): return "class-foo"
def bar():        return foo()          # ← 모듈 foo 호출
```
- 산출 엣지: `app.m.bar -> app.m.C.foo` (**틀림**). 정답 `app.m.bar -> app.m.foo` 는 **부재**.
- `bar->app.m.foo present: False` / `bar->app.m.C.foo present: True` 로 확인.

**왜 비차단 불가(§2B 필수질문 "거버넌스 목적에 직접 구멍을 내나?" → 예)**: 모듈함수 `foo` 가 sink(frozen 존/`@gov sink`)일 때, `bar` 는 실제로 `foo` 를 호출하는데 엣지가 `C.foo` 로 새어나가 **sink `foo` 의 역도달(TASK-024) 상류에서 `bar` 가 누락**된다 → `bar` 변경이 sink 에 도달하는데도 **미발화 = 민감 변경 포착 실패**. 이것은 거버넌스 하류에 직접 구멍이다. 게다가 **CLAUDE.md §2B 고정 적대 입력 세트가 "동명 오버로드"를 매 리뷰 필수·상설 회귀 픽스처로 명령**하는데, 제출 픽스처엔 이 케이스(모듈함수 vs 동명 클래스메서드)가 **없다**(conditional.py 는 동일-id 조건부 def 만 검증).
AC #1(호출을 repo 함수로 해소 — 그런데 **틀린** 함수로) · AC #4(동명은 **보수적 합집합** — 그런데 **하나만·틀린 것**으로 pick) 둘 다 이 케이스에서 불충족.

**보정(택1, ①권장)**:
1. **`visible_local_names` 의 클래스 확장 루프 제거**(2줄). Claude 가 사본 실증: `adv1` → `app.m.bar -> app.m.foo` 로 교정되고, **공식 callgraph fixture 엣지 7개는 원본과 바이트 동일**(이 루프는 정답 엣지에 대해 load-bearing 0). 클래스메서드는 원래도 `C.foo()`/`self.foo()` 로는 미해소(unresolved 노출)라 이 제거는 **정합·더 정직·더 보수적**.
2. 클래스메서드 해소를 유지하려면 **bare 이름이 형제 클래스메서드에 절대 바인딩되지 않도록 스코프 조건**을 걸고, 동명 충돌은 **pick-smallest 대신 union**(AC #4 정합)으로. (①보다 큰 변경 — 권장 안 함.)
3. **상설 회귀 픽스처 신설(§2B 명령)**: 같은 모듈에 모듈함수 `foo` + 클래스 `C.foo` + `foo()` 호출자 → 엣지가 **모듈 `foo` 로** 가고 `C.foo` 는 spurious callee 가 아님을 단언. (§2B "동명 오버로드" 적대세트 상설화.)

## 🟡 O-1 (비차단 관찰) — 중첩 데코레이터/기본인자 호출 조용한 유실

**위치**: `CallVisitor.visit_function()` (190~194행) — `for child in node.body` 만 방문, `node.decorator_list`·`node.args` 기본값은 미방문.

**관찰**: 함수 caller 가 명확한 **중첩** 정의의 데코레이터/기본인자 호출이 통째로 유실된다. fresh 실증 `adv4`:
```python
def make_wrapper(fn): return fn
def outer():
    @make_wrapper          # ← 실행 caller = outer, make_wrapper = repo 함수
    def inner(): return 1
    return inner()
```
→ 엣지 `[outer->outer.inner]` 만, `outer->make_wrapper` **부재** · `unresolved_calls` 에도 **없음**(조용한 유실). AC #2 "조용한 누락 금지" 와 약한 긴장.

**왜 비차단**: (a) 모듈-스코프 데코레이터/기본인자는 함수 caller 자체가 없어 함수-콜그래프상 정당 out-of-scope, (b) 중첩 케이스는 좁고, (c) R-1 과 달리 **틀린 엣지를 만들지 않고 누락만** 한다. 다만 정직성(D4) 관점에서 **TASK-025 고정 적대 입력 세트에 데코레이터/기본인자 항목 추가**로 coverage 노출 or 명시 한계로 고정 권장(강제 아님). R-1 재제출과 **묶어 처리해도 좋고 TASK-025 로 이월해도 됨** — 이월 시 본 관찰을 TASK-025 AC 근거로 남긴다.
