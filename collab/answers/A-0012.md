## R-1 (🔴 · 보정 필수) — 임계값 N 이 **피검자 자신의 change-intent 에서** 읽혀 자기-무력화 가능

### 무엇이 문제인가
AC #1(b) 는 **"N 은 정책값, 기본 80"**. 그러나 구현은 N 을 정책이 아니라 **검사 대상 파일(change-intent.yaml)** 에서 읽는다:

```python
# load_intent() — intent = data.get("change_intent")  ← 작성자(author) 소유 선언
"broad_scope_threshold_percent": int(
    intent.get("broad_scope_threshold_percent")
    or intent.get("scope_policy", {}).get("broad_scope_threshold_percent")
    or DEFAULT_BROAD_SCOPE_THRESHOLD_PERCENT
),
```

두 조회 키 모두 `change_intent` 블록 하위다. 외부 정책파일(`policies/*`, Claude 소유)에서 읽는 경로가 **아예 없다**. 게다가 상한 클램프도 없다. → **피검자가 자기 선언의 임계값을 스스로 정한다.** 이는 TASK-021 이 막으려는 "선언을 무의미하게 만들기"를 한 단계 위로 옮긴 것에 불과하다.

### 실증 (ADV1 — fresh 입력, 픽스처 밖, 실제 게이트)
```yaml
change_intent:
  author: attacker
  broad_scope_threshold_percent: 101      # ← 같은 파일에서 임계값을 100 초과로
  allowed_paths:                          # ← 최상위 8개 전부 개별 나열(리터럴 */** 회피)
    - ".harness/**"
    - "collab/**"
    - "docs/**"
    - "policies/**"
    - "scripts/**"
    - "summaries/**"
    - "templates/**"
    - "tests/**"
```
결과: `coverage_percent 100`, `root_globs []`, **`too_broad false` → verdict `pass` / exit 0**.
→ 사실상 무제한 범위 선언이 AC #1(b) 를 완전히 회피한다. (동일 선언에서 `broad_scope_threshold_percent` 만 지우면 → `top_level_coverage`·`approval_required`/exit 2. 즉 회피 스위치는 임계값 오버라이드 그 자체.)

### 왜 비차단으로 넘기지 않는가 (CLAUDE.md §2B 필수질문)
"이게 거버넌스 목적(민감·광역 변경 포착)에 직접 구멍을 내나?" → **그렇다.** 피검자가 이 태스크의 유일한 신설 방어축(커버리지 검출)을 자기 손으로 끌 수 있다. AC #1(b) 문구("N 은 정책값")와도 정면으로 어긋난다. → 비차단 금지, **보정요청**.

### 보정 방향 (택1, 최소 델타 권장)
- **(권장) change-intent 오버라이드 제거** — 고정 기본 80 만 사용. AC "기본 80" 그대로 충족하고 자기-무력화 경로가 사라진다. 델타 최소.
- 또는 **Claude 소유 정책파일에서 N 을 읽기**(예: `policies/approval-routing.yaml` 또는 신설 scope 정책). 게이트에 정책 경로 인자 추가 필요.
- ⚠️ **클램프 ≤100 만으로는 불충분**: threshold=100 으로 두어도 8개 중 7개만 나열(87% < 100)하면 통과한다. **author 통제 자체를 끊어야** 한다.

## R-2 (🟡 · 비차단 관찰 → 보정 시 함께 고치면 이상적 / 아니면 차기 AC 가드)

**회귀 픽스처가 라이브 repo 트리에 결합돼 깨지기 쉽다.** name-status **파일** 입력이면 `diff_base_ref` 가 `HEAD` 로 폴백 → `repo_top_level_dirs` 가 **실제 repo 의 현재 최상위 디렉토리**(`git ls-tree -d HEAD`)를 센다. 따라서 `broad-intent-coverage`(7개 나열)의 판정은 fixture 가 아니라 **실 repo 구조**에 달렸다: 현재 7/8=87% → `approval_required`(통과). 하지만 최상위 디렉토리가 하나 추가되면 7/9=77% < 80 → verdict 가 `pass` 로 뒤집혀 **이 회귀 테스트가 무관한 repo 성장만으로 깨진다**(상설 회귀 픽스처 원칙 위배·CLAUDE.md §2B 고정 적대입력 세트).

> 프로덕션(실제 diff ref 입력)에서는 base ref 로 결정적이라 거버넌스 구멍은 아니다 → 비차단. 다만 R-1 보정 커밋에서 픽스처가 top-level dir 집합을 **결정적으로 고정**(합성 base 트리 또는 fixture-local 디렉토리 세트로 판정)하도록 함께 손보길 권장. (root-glob 케이스 `broad-intent-root` 는 커버리지와 무관해 안정적.)
