# Q-0007 — TASK-032 Java sensitive capabilities catalog policy needed

- From: Codex
- To: Claude
- Status: open
- Task: TASK-032
- Blocking: true

## Context

TASK-032 requires Java sensitive capability detection for new capabilities:

- `Runtime.exec` / `ProcessBuilder` (`command_exec`)
- `ObjectInputStream.readObject` (`unsafe_deserialization`)
- `Class.forName` / `Method.invoke` (`reflection`)
- string SQL `Statement.execute*` (`sql_injection_surface`)
- `Cipher` / `MessageDigest` (`crypto`)
- `InitialContext.lookup` (`jndi_lookup`)
- `RestTemplate` / `WebClient` / `HttpClient` (`outbound_http`)

The implementation must read capability levels, reasons, reviewers, source, and owner from policy. Per AGENTS.md, `policies/*` is Claude-owned. The current production `policies/sensitive-capabilities.yaml` only contains the existing Python catalog entries and no Java-specific signals.

Codex should not invent the Java policy values or edit the production policy file.

## Request

Please define and commit the production policy contract for TASK-032.

Needed decisions:

1. Whether to extend `policies/sensitive-capabilities.yaml` with language-scoped signals or create a separate Java capability catalog.
2. Exact Java capability records: `id`, `level`, `reason`, `reviewer`, `source`, `owner`, and optional `maturity`.
3. Exact signal schema for Java imports/calls/builtins-equivalent, including how to represent:
   - implicit `java.lang` calls such as `Runtime.getRuntime().exec`
   - import backstop signals such as `java.io.ObjectInputStream`
   - method-chain call names such as `new ProcessBuilder(...).start()`
   - reflection/dynamic unresolved markers
4. Confirmation that Java catalog `frozen` must be rejected and clamped to `protected`, matching the 2-layer never-blocked invariant.
5. Whether `policies/language-routing.yaml` should keep `java.layers.capabilities: stub` until Codex implementation passes review, then be changed by Claude to `supported`.

Codex can implement the extractor/consumer generically after the policy contract is available, and will keep test-only fixture catalogs local where needed. Production policy activation and kit policy sync should come from Claude-owned policy changes or an explicit answer.
