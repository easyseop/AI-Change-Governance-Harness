# H-0001 — 자기거버넌스 zone(`.harness/**`·`tests/*`) 변경의 머지 권한: 현행 해석 확인 요청

- From: Claude   To: 사람(형)
- Status: open
- 성격: **비차단 확인 요청** — 진행 중인 머지를 막지 않는다(A-0044/D-088 은 현행 해석대로 이미 머지).

## 배경 — 왜 형의 판단이 필요한가

`policies/sensitive-zones.yaml` 은 TASK-018(D-028 · AC#6 자기보호)에서 이 하네스 **자신의 집행 경로**를
`protected` 로 등록했다:

```
- path: "policies/**"        level: protected  required_approval: security-reviewer
- path: ".harness/**"        level: protected  required_approval: security-reviewer
- path: "tests/run-tests.sh" level: protected  required_approval: security-reviewer
- path: "tests/cases.yaml"   level: protected  required_approval: security-reviewer
```

그래서 이 repo 에 자기 게이트를 돌리면 **Codex 의 모든 dev 게이트 태스크**가 다음 카드를 낸다:

```
verdict: approval_required · protected_touched: .harness/gates/… · tests/cases.yaml · tests/run-tests.sh
required_approval: security-reviewer
```

반면 `CLAUDE.md` §3·D-007 의 "민감 변경(🔴🟠 = 정산·인증/인가·암호화·DB migration·infra)" 은
**도메인 기준** 목록이라 여기 해당하지 않는다. 두 기준이 서로 다른 답을 준다.

지금까지의 **실제 운영은 후자**였다 — D-074·D-078·D-080·D-082·D-085 에서 Claude 가
`.harness/gates/*` 변경을 리뷰 통과 후 직접 `main` 머지했다. 하지만 이 해석은 **어디에도 명문화돼 있지 않다.**

**사실 정정도 하나 있다**: D-085(TASK-035)에 Claude 가 적은 "카드 `protected_touched` 없음" 은 **틀렸다**.
TASK-035 도 `.harness/gates/*`·`tests/cases.yaml`·`tests/run-tests.sh` 를 건드렸으므로 같은 카드가 나왔어야 한다.
즉 그동안 **카드를 정확히 확인하지 않은 채** 비민감 판정을 해 온 구간이 있다.

## 선택지

- **A (현행 유지 · Claude 추천)** — repo 내부에서는 자기거버넌스 zone 의 `security-reviewer` 역할을
  **Claude 의 적대적 리뷰 + 구현자≠머지자(D-007)** 가 수행하는 것으로 본다. D-007 의 "민감" =
  CLAUDE.md §3 **도메인 목록**. 대신 `CLAUDE.md`/`collab-protocol.md` 에 **한 줄 명문화**하고,
  리뷰 기록에 카드의 `protected_touched` 를 **정직하게 적는다**(A-0044 §5 방식).
  - 근거: 이 zone 의 원래 목적은 **도입 조직의 CI** 에서 "게이트 편집엔 사람 승인"을 강제하는 것.
    개발 repo 자신에는 상호견제라는 더 강한 통제가 이미 걸려 있다.
- **B (엄격)** — `.harness/**` 를 건드리는 모든 태스크는 형 승인 후 머지. 하네스 자기 코드에도
  자기 정책을 문자 그대로 적용.
  - 비용: **거의 모든 Codex 태스크가 형 승인 대기**가 된다(TASK-037·038 포함).
- **C (분리)** — zone 을 쪼갠다. 예: `.harness/gates/**` 중 **판정 로직**(`check-*.py`)은 형 승인,
  **추출기**(`extract-*.py`)·테스트는 Claude 머지. 판정 완화 위험이 실제로 큰 쪽만 사람에게.

## 영향

- **금전**: 없음.
- **보안**: A/C 는 "게이트를 무력화하는 편집"의 최종 방어선이 Claude 리뷰 1인. B 는 사람 2중 확인.
  단 현재도 `check-policy-change`(정책 완화·집행 우회 탐지)가 별도로 돌고 매 리뷰 rig-and-revert 로
  가드 load-bearing 을 검증하고 있다.
- **운영**: B 는 처리량이 크게 떨어진다(형 대기 시간이 병목). C 는 zone 정의 수정 필요.
