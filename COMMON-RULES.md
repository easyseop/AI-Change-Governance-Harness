# COMMON-RULES.md — 공통 규칙 (Claude·Codex 둘 다 필독)

> 🔴 **필독**: 새 세션이면 `START-HERE.md` → 자기 역할 파일(`CLAUDE.md`/`AGENTS.md`) → **이 파일** 순으로 읽어라.
> 이 파일은 두 모델(Claude·Codex)에 **공통으로 적용**되는 규칙이다. 역할별 규칙은 각 역할 파일에 있다.
> **소유: 사람(형).** 두 AI는 읽고 따른다(읽기 전용). 규칙을 바꾸자는 제안은 `collab/needs-human/` 로 올린다.

---

## 0. 시스템 맥락 — 우리가 무엇을 지키는가 (은행 운영 관점)
이 하네스가 거버넌스하는 대상은 **은행 운영 시스템 전반**이다. 코어뱅킹만 있는 게 아니라, 요즘 은행은
일반 IT 기업과 똑같이 **데이터 플랫폼·AI/Agent 플랫폼**까지 개발/운영한다. 변경을 평가할 땐 늘 이 지형을 떠올린다.

| 도메인 | 예 | 기본 리스크 결 |
|---|---|---|
| 코어뱅킹/계정계 | 정산·이자·자금이체·원장 | 🔴 금전 직접영향, 되돌리기 어려움 |
| 채널/대외/보안 | 인증·인가·암호화·API GW | 🟠 보안경계, 사람검토 필수 |
| 데이터 플랫폼 | ETL·파이프라인·DW·피처스토어 | 🟠 PII·정합성이면 승인요구 / 그 외 🟡 |
| AI/Agent 플랫폼 | 모델서빙·오케스트레이션·프롬프트·툴 | 🟠 비결정성·외부호출·권한확대 |
| 일반 IT | 내부도구·UI·문서·일반로직 | 🟢 저위험 |

→ **"은행 = 코어만 민감"이 아니다.** 데이터/AI 플랫폼의 *데이터 유출·권한 확대·외부 의존*도 동급으로 본다.

## 1. 보수적 개발 원칙 (둘 다 지킨다 / Claude 는 이걸로 평가한다)
**핵심: 요청한 부분만 고치고, 그 외 코드는 최대한 건드리지 않는다.**
- 요청 범위(change-intent) **밖 파일/경로 수정 금지.** 필요하면 먼저 묻는다(`collab/questions/` 또는 `needs-human/`).
- 무관한 **리팩터링·이름변경·포맷팅·임포트 정리 금지** (diff 노이즈 = 리뷰 비용·리스크 증가).
- 기존 **동작/공개 인터페이스 보존.** 부수효과 최소화.
- 공유/공통 모듈은 **꼭 필요할 때만**, 영향범위(blast radius)를 명시하고 보수적으로.
- diff 는 작게. 파일 단위로 자문: *"이 변경이 목표 달성에 꼭 필요한가?"*

**Claude 리뷰 평가축(보수성):** 요청 외 파일을 건드렸나 / 무관한 변경(리팩터·포맷)이 섞였나 /
blast radius 가 의도보다 큰가 → 있으면 `scope-creep`·`over-reach` 로 감점·보정요청.

## 2. 브랜치 전략 (단일 브랜치 금지 · 날짜별 생성)
- **하나의 장수(long-lived) 작업 브랜치를 쓰지 않는다.** 작업 단위마다 **날짜별 브랜치**를 새로 판다.
- 형식: `<역할>/<YYYY-MM-DD>-<주제슬러그>`
  - 예) `codex/2026-06-28-task001-change-intent`, `claude/2026-06-28-review-task001`
- base 는 **항상 최신 `main`.** 하루에 작업이 여럿이면 주제별로 브랜치를 나눈다.
- `main` **직접 push 금지.** 리뷰 통과 후 머지(자기 머지 금지 — §안전장치).

---

## 3. 커밋 & 푸시 규칙 (상세히 적는다)
한 줄 요약만 남기지 않는다. **무엇을·왜·무엇을 고쳤는지** 본문에 적는다.
```
<type>: <한 줄 요약 (무엇을)>

왜(목적):   이 변경이 해결하는 요구/문제
무엇을 수정: 건드린 파일/영역과 핵심 변경점
무엇을 고침: 버그/이슈를 고쳤다면 증상→원인→해결 (없으면 '신규')
영향 범위:  blast radius, 호환성/마이그레이션 주의
관련:       TASK-00X / D-00X / Q-XXXX / 요구사항(R-XXX)
```
- type 예: `feat` / `fix` / `docs` / `refactor` / `test` / `policy` / `chore`.
- 푸시는 자기 **날짜별 브랜치**로. 푸시할 때마다 §4 요약을 갱신한다.

---

## 4. 요약 폴더 (`summaries/`) — 푸시마다 누적
- 날짜별 파일: `summaries/YYYY-MM-DD.md`. 그 날 작업을 **누적**으로 쌓는다(덮어쓰기 금지, 아래에 추가).
- 푸시할 때마다 한 항목 추가: **무엇을 위해 / 무엇을 수정 / 무엇을 고침 / 브랜치·커밋**.
- 형이 날짜 파일만 보면 그날 무슨 일이 있었는지 다 보이게. (형식: `summaries/README.md`)

---

## 5. 사용자 판단 필요 (`collab/needs-human/`) — 형에게 올리는 큐
AI끼리 못 정하거나 **형의 결정/승인이 필요한 것**은 여기에 모은다.
- 항목 파일: `collab/needs-human/H-XXXX.md` (`Status: open|resolved`).
- 올리는 경우: 조직 정책값(🟡), 민감변경 최종 승인, 규칙 변경 제안, AI간 합의 불가(재오픈 ≥3회),
  보안/금전 영향 의심, 요청 범위를 넘어야만 하는 상황.
- 형이 결정하면 `resolved` 로 바꾸고, 확정사항은 `collab/decisions.md` 로 승격.
