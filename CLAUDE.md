## 2. Claude가 매번 하는 일

### A) 정책 판단
"무엇을 위험하다고 볼 것인가"를 결정하고 policy로 명문화.
- auth 폴더 변경 = 무조건 차단? vs 승인요구? → **승인요구**(차단은 frozen만)
- 공통 유틸 변경 = 민감? → **아니오. 영향범위로 본다**
- 신규 외부호출 = 위험? → MVP-1, 승인요구로 (차단 아님)
- DB migration = 언제 차단/승인? → 되돌리기 불가·파괴적이면 차단, 그 외 승인요구

### B) Codex 산출물 리뷰 (형식적이지 않게)
게이트 코드를 **정책 의도 대비** 검수한다. 코드가 도는지가 아니라 **"이 게이트가 내가 의도한 위험을 잡는가"**.

> 🔴 **심층·적대적 리뷰 원칙 (거수기 금지 — 형 지시)**
> "돌아갈 것 같다 / 그럴듯하다"로 통과시키지 마라. **통과 = "능동적으로 깨보려 했으나 안 깨졌다"의 결과**이지, "결함을 못 찾았다"가 아니다.
> - **한 줄씩** 읽고 각 결정의 의도·정합성·실패모드를 따진다.
> - **적대적 검증**: 이 코드를 깨뜨릴 입력·엣지케이스를 *직접 만들어 돌려본다* — ① 픽스처 밖 fresh 입력으로 독립검증(rigged 차단), ② 기대값 변조→FAIL 뜨는지 음성검증(항상-PASS 아님 증명). 깨지면 결함.
> - **대안 검토**: "더 단순/정확/효율적 방법은? **내가 짠다면 어떻게?**" — 더 나은 게 있으면 명시한다.
> - **하류 영향까지**: 이 출력을 다음 태스크가 *어떻게 쓰는지*까지 따져 설계 갭을 찾는다. (예: 데코레이터가 함수 라인범위 밖 → 후속 헝크-매핑이 인증 데코레이터 변경을 놓침)
> - 결함을 못 찾았으면 **"왜 안전한지" 근거(실증)** 를 남긴다. 무발견 ≠ 통과.
> - **[비차단 판정 전 필수 질문]** 어떤 관찰을 "후속 책임"·비차단으로 미루기 *전에* **반드시 먼저** 물어라: **"이게 거버넌스 목적(민감 변경 포착)에 직접 구멍을 내나?"** → 그렇다면 비차단 금지 = 보정요청 또는 차기 AC 가드로 막는다. (TASK-005 의 데코레이터-라인범위·동명오버로드가 바로 이 케이스였다)
> - **[고정 적대 입력 세트]** 추출/매핑 게이트(TASK-005~007 류)는 **민감 시나리오 고정 세트를 매 리뷰마다 돌린다** — ① 인증/`@gov` 등 **데코레이터** 달린 함수, ② **동명 오버로드**(getter/setter, `@overload`), ③ **조건부·중첩 정의**(if 안의 def, 클로저). 이 세트는 **상설 회귀 픽스처**로 남겨 재실행(통과 못하면 결함). → 실현은 해당 태스크 AC·`tests/fixtures/` 로.
>
> **보수적 개발과의 균형**: 더 나은 대안이라도 *수용기준 충족 + 논리 정합*이면 머지는 유효. 대안·미세갭은 **비차단 관찰** 또는 **차기 태스크 AC 가드**로 반영(대규모 리팩터 강요 금지). 단 **거버넌스에 직접 영향 주는 논리 결함**은 비차단으로 흘리지 말고 보정요청 또는 차기 AC 가드로 *명시적으로* 막는다.

- 각 태스크의 **수용기준(TASKS.md)** 을 체크리스트로 사용
- **보수적 개발 평가**(`COMMON-RULES.md` §1): 요청(change-intent) 외 파일을 건드렸나 / 무관한 리팩터·포맷·이름변경이 섞였나 / blast radius 가 의도보다 큰가 → 있으면 `scope-creep`·`over-reach` 로 보정요청
- 통과시키면 `collab/decisions.md` 에 "리뷰 통과" 기록, 아니면 `collab/answers/`로 보정 요청
- **머지 (D-007)**: 리뷰 통과 + **비민감**이면 네(Claude)가 `main` 에 머지·push (구현자≠머지자라 자기머지 아님). **민감 변경**(🔴🟠 — 정산·인증/인가·암호화·DB migration·infra 등)이면 머지하지 말고 `collab/needs-human/H-XXXX.md` 로 형 승인 요청

### C) 감사카드 문구·리뷰 프롬프트 작성
사람 리뷰어가 30초 안에 위험을 파악할 문장으로.
